# ADR-005: Хранение API-ключей и чувствительных данных

**Статус:** Proposed  
**Дата:** 14 июля 2026  
**Контекст:** product-foundation.md §21 («внешние API-ключи не хранятся открытым текстом»), §8 (LLM providers), §6 (принцип local-first), ADR-004 (desktop shell)

---

## Проблема

Пользователь подключает облачные LLM-провайдеры (OpenAI, Anthropic, OpenRouter и т.д.) по API key. Также могут требоваться ключи для облачных генераторов изображений, embedding endpoints и внешних сервисов.

Метрика качества §21 явно требует: **«внешние API-ключи не хранятся открытым текстом»**.

Необходимо решить:

1. **Где и как хранить** API-ключи и другие секреты (tokens, passwords).
2. **Как обеспечить безопасность** в различных режимах работы:
   - **Desktop single-user**: Tauri shell, один пользователь на локальной машине.
   - **LAN-режим**: backend доступен по сети, другие устройства отправляют запросы.
   - **Docker/self-hosted**: backend запущен как контейнер.
3. **Как обеспечить переносимость**: что происходит при миграции кампании на другой ПК.
4. **Нужна ли аутентификация** между backend и локальными inference серверами (Ollama, ComfyUI, KoboldCpp).

### Типы секретов

| Секрет | Формат | Частота доступа | Чувствительность |
|--------|--------|----------------|-----------------|
| Cloud LLM API key | Строка 40–100 символов | Каждый ход (каждый LLM call) | Высокая (деньги) |
| Embedding endpoint key | Строка | Каждый memory update | Высокая |
| Image gen API key | Строка | По запросу | Средняя |
| Ollama URL/порт | URL | Каждый ход | Низкая (localhost) |
| ComfyUI URL/порт | URL | По запросу | Низкая (localhost) |
| LAN access token | Строка/JWT | Каждый HTTP запрос в LAN | Средняя |

## Рассмотренные варианты

### Вариант A: OS Keychain (Windows Credential Manager / macOS Keychain / Linux libsecret)

Использовать нативные хранилища секретов операционной системы через Tauri plugin или Python-библиотеку (`keyring`).

**Механизм:**

- На **Windows**: Windows Credential Manager (DPAPI encryption, привязан к учётной записи пользователя).
- На **macOS**: Keychain Services (AES-256, привязан к пользователю, опционально к приложению).
- На **Linux**: libsecret / GNOME Keyring / KDE Wallet (D-Bus API, требует работающий keyring daemon).

**Доступ:**

- Из **Tauri (Rust)**: `tauri-plugin-os` или `keyring` crate.
- Из **Python**: библиотека `keyring` (кроссплатформенная обёртка над нативными keychain).
- Хранение: `service = "personaldm"`, `username = "openai_api_key"`, `password = <ключ>`.

**Плюсы:**

- **Максимальная безопасность** из всех вариантов. Ключи зашифрованы на уровне ОС, доступны только текущему пользователю.
- **Не требует пароля** от пользователя — ОС управляет шифрованием.
- **Стандартный подход**: так хранят секреты VS Code, Docker Desktop, Slack, 1Password и другие desktop-приложения.
- **Защита при физическом доступе**: ключи зашифрованы учётными данными ОС, не извлекаемы без входа в систему.
- **Не входит в campaign bundle**: при экспорте кампании ключи не утекают.

**Минусы:**

- **Непереносимость**: ключи привязаны к учётной записи на конкретном ПК. При миграции на другой компьютер пользователь должен ввести ключи заново. Это минус UX, но плюс безопасности.
- **Linux headless**: на серверах без GUI (Docker, headless Linux) GNOME Keyring / KDE Wallet могут не быть установлены. `keyring` библиотека падает или использует plaintext fallback.
- **Docker/self-hosted**: в контейнере OS Keychain недоступен. Нужен отдельный fallback для этого режима.
- **LAN-режим проблема**: ключи хранятся на host-машине в keychain. Другие устройства в LAN обращаются к backend по HTTP. Backend извлекает ключи из keychain при каждом LLM-вызове — это работает, но если backend скомпрометирован по сети, ключи доступны через API. Нужна аутентификация самого backend.
- **Зависимость от Tauri**: если используем Tauri plugin для keychain, в web-only режиме (Фаза 1 по ADR-004) keychain недоступен из браузера.
- **Python `keyring`**: на некоторых Linux-дистрибутивах требует ручной настройки backend (установка `gnome-keyring` или `kwallet`).

**UX flow:**

```
Пользователь → Settings → API Keys → вводит ключ OpenAI
  → Frontend → Tauri invoke / HTTP POST → Backend
    → keyring.set_password("personaldm", "openai_api_key", "<key>")
    → ОС шифрует и сохраняет
    
При LLM call:
  → Backend → keyring.get_password("personaldm", "openai_api_key")
    → ОС расшифровывает и возвращает
    → Backend использует для API call
```

### Вариант B: Encrypted config file (AES-256)

Хранить секреты в зашифрованном файле на диске. Ключ шифрования получается из:
- (B1) OS Keychain (гибрид с Вариантом A);
- (B2) Пароля пользователя (PBKDF2/Argon2 → AES key);
- (B3) Machine-specific key (MAC address, hardware ID — слабый вариант).

**Механизм:**

```
secrets.enc = AES-256-GCM(
  key = PBKDF2(user_password, salt) | keychain_derived_key,
  data = JSON({
    "openai_api_key": "sk-...",
    "anthropic_api_key": "sk-ant-...",
    ...
  })
)
```

**Плюсы:**

- **Переносимость** (при варианте B2): файл `secrets.enc` можно скопировать на другой ПК, расшифровать паролем пользователя. Ключи мигрируют вместе с кампанией (если пользователь этого хочет).
- **Работает везде**: не зависит от наличия OS Keychain. Работает в Docker, headless Linux, любой ОС.
- **Единый механизм**: один подход для всех платформ, нет платформозависимого кода.
- **Контроль**: можно включить файл в backup или исключить из него — выбор пользователя.
- **Гибрид B1**: ключ шифрования в keychain — получаем безопасность keychain + переносимый формат файла.

**Минусы:**

- **B2 (пароль)**: пользователь должен вводить пароль при каждом запуске приложения. Для local-first приложения с одним пользователем это лишний шаг. Альтернатива: кэшировать расшифрованные ключи в RAM на время сессии.
- **B2 (пароль)**: если пользователь забудет пароль, ключи потеряны (их можно ввести заново, но это UX-потеря).
- **B3 (machine key)**: security through obscurity — слабая защита, machine ID можно извлечь.
- **Реализация**: нужно самостоятельно реализовать криптографию (или использовать `cryptography` библиотеку). Ошибка в реализации = утечка. Это не rocket science, но ответственность на нас.
- **Файл на диске**: при неправильных permissions (например, на shared Linux) файл может быть прочитан другим пользователем. Нужно устанавливать `chmod 600`.
- **Hot keys in memory**: после расшифровки ключи лежат в RAM Python-процесса в plaintext. При dump'е памяти (core dump, swap) они могут утечь. Это общая проблема всех вариантов.

**UX flow:**

```
Первый запуск:
  → Приложение предлагает установить master password (или использует keychain для ключа шифрования)
  → Создаёт secrets.enc с пустым JSON

Добавление ключа:
  → Settings → ввод ключа → расшифровка secrets.enc → добавление → перешифровка → сохранение

Запуск:
  → Приложение запрашивает master password (или извлекает ключ из keychain)
  → Расшифровывает secrets.enc в RAM
  → Ключи доступны на время работы
```

### Вариант C: Environment variables (dotenv)

Хранить секреты в `.env` файле. Backend загружает их через `python-dotenv` или `pydantic-settings`.

**Механизм:**

```bash
# .env
OPENAI_API_KEY=sk-...
ANTHROPIC_API_KEY=sk-ant-...
COMFYUI_URL=http://localhost:8188
OLLAMA_URL=http://localhost:11434
```

```python
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    openai_api_key: str | None = None
    anthropic_api_key: str | None = None
    
    class Config:
        env_file = ".env"
```

**Плюсы:**

- **Простота**: стандартный подход для Python-сервисов. Все разработчики знакомы.
- **Docker-friendly**: `docker run -e OPENAI_API_KEY=... ` или `docker compose` с `.env`.
- **Нет зависимостей**: `python-dotenv` — минимальная библиотека. `pydantic-settings` уже в стеке.
- **Переносимость**: `.env` файл можно скопировать на другой ПК.
- **12-factor app**: соответствует стандартному подходу для конфигурации.

**Минусы:**

- **Открытый текст**: `.env` файл содержит ключи в plaintext. Это прямо нарушает требование §21.
- **File permissions**: единственная защита — `chmod 600` на `.env`. На Windows — ACL. Пользователи часто не настраивают это.
- **Git risk**: `.env` файл может случайно попасть в git (даже при наличии `.gitignore`). Для open-source продукта — реальный риск пользовательских форков.
- **Видимость в процессах**: env vars видны через `/proc/<pid>/environ` на Linux, через Process Explorer на Windows.
- **Не desktop-friendly UX**: пользователь должен вручную редактировать текстовый файл. Нет GUI для управления ключами без дополнительной разработки.
- **LAN-режим**: если `.env` доступен через file share или если backend скомпрометирован — ключи утекают.
- **Не подходит для non-technical пользователей**: «откройте файл .env в текстовом редакторе» — это не UX desktop-приложения.

### Вариант D: Встроенный vault (SQLCipher encrypted DB для secrets)

Использовать отдельную SQLCipher базу данных для хранения секретов. SQLCipher — это SQLite с AES-256 шифрованием «на лету».

**Механизм:**

```python
import sqlcipher3

conn = sqlcipher3.connect("secrets.db")
conn.execute("PRAGMA key = 'user_master_password_or_derived_key'")
conn.execute("CREATE TABLE IF NOT EXISTS secrets (key TEXT PRIMARY KEY, value TEXT)")
conn.execute("INSERT OR REPLACE INTO secrets VALUES ('openai_api_key', ?)", [api_key])
```

**Плюсы:**

- **Единая технология**: SQLite уже в стеке. SQLCipher — расширение SQLite. Знакомый API.
- **Шифрование на уровне страниц**: весь файл зашифрован, включая метаданные. Нет возможности частично прочитать.
- **ACID**: транзакционность, crash recovery — не потеряем ключи при сбое.
- **Переносимость**: один файл `secrets.db`, зашифрованный. Можно скопировать на другой ПК, расшифровать тем же паролем.
- **Расширяемость**: можно хранить не только ключи, но и метаданные (дата добавления, провайдер, последнее использование, валидность).
- **Работает везде**: SQLCipher кроссплатформенный, не зависит от OS Keychain.

**Минусы:**

- **Зависимость**: `sqlcipher3` — compiled extension, требует `libsqlcipher`. На Windows — предсобранные wheels. На Linux — `apt install libsqlcipher-dev`. Усложняет bundling.
- **Master password**: те же UX-проблемы, что у Варианта B2 — пользователь должен вводить пароль.
- **Over-engineering**: для хранения 3–5 строк (ключей) полноценная зашифрованная БД — это как «подчинять Древнего Демона ради уборки комнаты». Файл + AES (Вариант B) проще.
- **Два SQLite**: основная БД кампании (SQLite) + секреты (SQLCipher). Две разные библиотеки, потенциальные конфликты.
- **Performance**: SQLCipher добавляет ~5–15% overhead на операции. Для 3–5 чтений при старте — несущественно, но зависимость тянет свой вес.

## Дополнительные аспекты

### LAN-режим: защита API-ключей от других устройств

Когда backend доступен по сети (`0.0.0.0:<port>`), другие устройства в LAN отправляют HTTP/WS запросы. Backend имеет доступ к расшифрованным ключам.

**Проблема**: любое устройство в LAN может:
- Вызывать LLM через backend (расходуя чужие API-кредиты).
- Потенциально извлечь ключи, если есть endpoint, который их возвращает (не должно быть, но нужно гарантировать).

**Решение (обязательно, независимо от выбора варианта):**

1. **Bearer token для LAN-сессий**: при активации LAN-режима backend генерирует одноразовый token (или QR-код). Клиент на другом устройстве вводит token для аутентификации. Все запросы требуют `Authorization: Bearer <token>`.
2. **Никогда не отдавать ключи через API**: endpoint `/api/settings` возвращает маску (`sk-...***abc`), а не полный ключ.
3. **Rate limiting**: ограничить количество LLM-вызовов с LAN-устройств.
4. **Явный opt-in**: LAN-режим не включён по умолчанию. Пользователь явно активирует его.

### Аутентификация между backend и локальными inference серверами

**Ollama:** по умолчанию не требует аутентификации (localhost). При доступе по сети — нет встроенной auth. Рекомендуется:
- Ограничить Ollama bind address до `127.0.0.1`.
- Если нужен доступ по сети — поставить reverse proxy с auth.

**ComfyUI:** аналогично — нет встроенной auth. Bind на `127.0.0.1`.

**KoboldCpp / vLLM / llama.cpp server:** аналогично.

**Рекомендация**: для localhost inference серверов аутентификация не требуется (они слушают только на `127.0.0.1`). Документировать, что при сетевом доступе к inference серверам пользователь должен сам обеспечить аутентификацию (nginx, Caddy и т.д.). Наш backend вызывает их по localhost — это безопасно.

### Что происходит при миграции на другой ПК

| Вариант | Миграция | Действие пользователя |
|---------|----------|----------------------|
| A: OS Keychain | Ключи не мигрируют | Ввести заново |
| B: Encrypted file | Файл мигрирует | Ввести master password |
| C: .env | Файл мигрирует | Скопировать .env (plaintext!) |
| D: SQLCipher | Файл мигрирует | Ввести master password |

## Рекомендация

**Рекомендуемая стратегия: многослойная, зависит от режима работы.**

### Уровень 1 (Desktop с Tauri — основной): Вариант A (OS Keychain) + B1 (encrypted file с keychain-derived key)

- API-ключи хранятся в **OS Keychain** через Python `keyring` или Tauri plugin.
- Для переносимости и backup: опциональный **encrypted config file**, ключ шифрования которого хранится в Keychain.
- Пользователь не вводит пароль — Keychain привязан к учётной записи ОС.
- При миграции: пользователь вводит ключи заново (или экспортирует/импортирует encrypted config file с master password).

### Уровень 2 (Web-only / Docker — ранний этап и self-hosted): Вариант B2 (encrypted file + master password)

- OS Keychain недоступен.
- При первом запуске пользователь устанавливает master password.
- `secrets.enc` хранит ключи, зашифрованные AES-256-GCM с ключом из Argon2(password).
- При каждом запуске backend запрашивает пароль (или принимает через env var `PDM_MASTER_PASSWORD` для Docker — при понимании рисков).

### Уровень 3 (LAN-режим — любой): Bearer token

- Backend генерирует session token при активации LAN-режима.
- Все запросы от внешних устройств требуют `Authorization: Bearer <token>`.
- Token отображается в UI / QR-код.

### Не рекомендуется

- **Вариант C (dotenv в plaintext)**: нарушает §21 и создаёт риск утечки. Допустим только для localhost URL-ов и несекретных настроек.
- **Вариант D (SQLCipher)**: over-engineering для текущего объёма секретов. Может быть пересмотрен, если количество секретов значительно вырастет (десятки провайдеров, пользовательские токены multiplayer).

## Последствия

1. **Зависимость от ADR-004**: в web-only фазе (Фаза 1) OS Keychain недоступен. Backend должен поддерживать encrypted config file как fallback с первого дня.
2. **UX Settings**: нужен GUI для ввода/редактирования API-ключей. Frontend показывает маскированные значения, позволяет заменить/удалить ключ.
3. **Python `keyring`**: нужно включить в зависимости и протестировать на Windows (WinCred), macOS (Keychain), Linux (GNOME Keyring + fallback).
4. **Startup flow**: при первом запуске — onboarding: «выберите LLM провайдер → введите API key (или используйте локальный Ollama)».
5. **Campaign export**: экспорт campaign bundle **не должен** включать ключи. Только структурные данные.
6. **Audit**: логировать, когда и каким провайдером были использованы ключи (без логирования самих ключей).
7. **Rotation**: UI должен позволять легко заменить ключ. При замене старый ключ немедленно удаляется из хранилища.
8. **Несекретные настройки** (Ollama URL, ComfyUI URL, theme, language): хранятся в обычном config file (JSON/TOML) без шифрования. Не смешивать с секретами.

## Связанные решения

- **ADR-004**: Выбор desktop shell влияет на доступность OS Keychain (Tauri plugin vs Python keyring vs недоступно).
- **§21 product-foundation**: «API-ключи не хранятся открытым текстом» — явное требование.
- **§6 product-foundation**: Принцип local-first. Ключи остаются на машине пользователя.
- **§8 product-foundation**: LLM providers — OpenAI-compatible adapter требует API key.
- **§7 product-foundation**: LAN-режим создаёт дополнительную поверхность атаки.
