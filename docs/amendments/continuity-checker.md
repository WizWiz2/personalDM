# Continuity Checker — Двухслойная Архитектура Валидации Канона

**Статус:** дополнение к product-foundation.md  
**Расширяет:** секция 9 (Логическая архитектура, компонент Continuity Checker), секция 11.3 (Валидация)  
**Версия:** 0.1  
**Дата:** 14 июля 2026

---

## 1. Преамбула

Continuity Checker — страж канона. Его задача — перехватить ответ Narrative LLM **до** или **сразу после** отправки пользователю и убедиться, что ответ не разрушает установленную реальность кампании.

В product-foundation.md (секция 11.3) перечислены проверки: доступность знаний, физическая непротиворечивость, состояние персонажей, владение предметами, локации, временная последовательность, конфликт с каноном, несогласованное изменение внешности, нарушение ruleset. Данный документ раскрывает **как именно** каждая проверка реализуется, когда запускается и что происходит при нарушении.

### Почему двухслойная архитектура

Не все нарушения канона одинаковы. Часть из них — жёсткие структурные противоречия, которые можно обнаружить запросом к базе данных за миллисекунды. Другая часть — нечёткие смысловые расхождения, которые способна уловить только языковая модель.

Попытка решить всё одним слоем неизбежно приводит к одному из двух провалов:

1. **Только SQL.** Быстро и дёшево, но пропускает нечёткие нарушения: персонаж внезапно изменил манеру речи, физически невозможное действие описано красивым языком, эмоциональная реакция противоречит установленному характеру. SQL не понимает семантику текста.

2. **Только LLM.** Ловит нюансы, но каждая проверка — это 500–2000 мс и токены. Запускать 8–13 LLM-вызовов синхронно на каждый ход — значит добавить 5–15 секунд к каждому ответу. Более того, LLM-проверки имеют precision ~80–90%: ложные срабатывания раздражают, а ложные отрицания опасны.

**Решение — два слоя:**

| Характеристика | Слой 1: Детерминированный (SQL) | Слой 2: Нечёткий (LLM) |
|---|---|---|
| Стоимость | 0 LLM-вызовов | 1–3 LLM-вызова |
| Латентность | < 50 мс | 500–5000 мс |
| Precision | 100% (по определению) | ~80–90% |
| Recall | Ограничен тем, что можно выразить SQL | Ловит семантические нарушения |
| Когда запускается | Синхронно, до отправки | Асинхронно, после отправки |
| Блокирует ответ | Да (при critical) | Нет |

Слой 1 — это быстрый, дешёвый и абсолютно надёжный фильтр. Он отсекает грубые нарушения (мёртвый NPC ведёт диалог, предмет одновременно в двух инвентарях) ещё до того, как пользователь увидит ответ. Слой 2 — это асинхронный глубокий анализ, который работает в фоне и предупреждает о тонких несоответствиях (персонаж ведёт себя не в характере, речь изменилась без причины).

## 2. Слой 1 — Детерминированные проверки (SQL-based)

### 2.1. Принцип работы

Каждая проверка Слоя 1 — это один или несколько SQL-запросов к **Relational Canon** (основному хранилищу структурированного состояния кампании). Проверки выполняются над **Proposed Delta** — набором изменений, которые Narrative LLM предлагает применить к состоянию мира после текущего хода.

Вход: Proposed Delta (структурированный JSON, описывающий изменения состояния) + текущее состояние Relational Canon.

Выход: список `ValidationResult`, каждый из которых содержит:

```json
{
  "check_id": "ITEM_OWNERSHIP_CONFLICT",
  "severity": "critical",
  "entity_ids": ["item_123", "char_456", "char_789"],
  "message": "Предмет 'Меч Зари' уже принадлежит персонажу 'Лиара'. Нельзя передать его 'Страж' без явного события передачи.",
  "suggestion": "Добавить событие передачи или убрать предмет из инвентаря текущего владельца.",
  "source_turns": [42, 67],
  "auto_fixable": false
}
```

### 2.2. Перечень проверок

---

#### CHECK-D01: Владение предметом (Item Ownership Conflict)

**Описание:**  
Предмет уже принадлежит другому персонажу или находится в другой локации. Narrative LLM упоминает, что персонаж использует, достаёт или передаёт предмет, но по данным Relational Canon этот предмет находится не у него.

**Severity:** `critical`

**SQL-логика:**

```sql
-- Проверка: предмет item_id принадлежит ли персонажу char_id?
SELECT i.id, i.name, i.owner_id, i.location_id, 
       c.name AS current_owner_name
FROM items i
LEFT JOIN characters c ON i.owner_id = c.id
WHERE i.id = :item_id
  AND i.campaign_id = :campaign_id
  AND (i.owner_id != :expected_owner_id 
       OR i.owner_id IS NULL);
```

Если запрос возвращает строки — нарушение. Предмет либо принадлежит другому персонажу, либо лежит в локации без владельца.

**Пример:**  
Narrative LLM генерирует: *«Гарет достал из сумки Кристалл Теней и направил его на дверь.»*  
Proposed Delta содержит: `use_item(char=Гарет, item=Кристалл Теней)`.  
Relational Canon: `Кристалл Теней.owner_id = Лиара.id` (Лиара получила его в сцене 12).  
Результат: **CRITICAL** — Кристалл Теней принадлежит Лиаре, а не Гарету.

---

#### CHECK-D02: Локация персонажа (Character Location Mismatch)

**Описание:**  
Персонаж упоминается как присутствующий или действующий в локации, которая не совпадает с его текущей локацией в Relational Canon.

**Severity:** `critical`

**SQL-логика:**

```sql
-- Проверка: персонаж находится в ожидаемой локации?
SELECT c.id, c.name, c.current_location_id,
       l_current.name AS actual_location,
       l_expected.name AS expected_location
FROM characters c
JOIN locations l_current ON c.current_location_id = l_current.id
JOIN locations l_expected ON l_expected.id = :expected_location_id
WHERE c.id = :char_id
  AND c.campaign_id = :campaign_id
  AND c.current_location_id != :expected_location_id;
```

**Пример:**  
Narrative LLM: *«В таверне Лиара подсела к Гарету и шепнула...»*  
Relational Canon: `Лиара.current_location_id = Монастырь.id`.  
Лиара сейчас в монастыре, а не в таверне.  
Результат: **CRITICAL** — персонаж телепортировался без события перемещения.

---

#### CHECK-D03: Присутствие при событии (Presence at Scene)

**Описание:**  
Proposed Delta регистрирует персонажа как участника или свидетеля события в текущей сцене, но этот персонаж не числится среди участников сцены (`scene_participants`).

**Severity:** `critical`

**SQL-логика:**

```sql
-- Проверка: персонаж является участником текущей сцены?
SELECT c.id, c.name, s.id AS scene_id, s.location_id
FROM characters c
CROSS JOIN scenes s
WHERE s.id = :current_scene_id
  AND c.id = :char_id
  AND c.campaign_id = :campaign_id
  AND NOT EXISTS (
    SELECT 1 FROM scene_participants sp
    WHERE sp.scene_id = s.id AND sp.character_id = c.id
  );
```

**Пример:**  
Текущая сцена: «Совет в Тронном Зале». Участники: Король, Советник, Страж.  
Narrative LLM: *«Сафира, стоявшая у колонны, вмешалась в спор.»*  
Сафира не числится в `scene_participants`.  
Результат: **CRITICAL** — персонаж действует в сцене, в которой не присутствует.

---

#### CHECK-D04: Мёртвый или отсутствующий персонаж действует (Dead/Absent Character Acts)

**Описание:**  
Proposed Delta содержит действие, реплику или решение персонажа, чей статус в Relational Canon — `dead`, `unconscious`, `imprisoned`, `absent`, `petrified` или аналогичный неактивный статус.

**Severity:** `critical`

**SQL-логика:**

```sql
-- Проверка: персонаж в активном состоянии?
SELECT c.id, c.name, c.status
FROM characters c
WHERE c.id = :char_id
  AND c.campaign_id = :campaign_id
  AND c.status IN ('dead', 'unconscious', 'imprisoned', 
                    'absent', 'petrified', 'banished',
                    'comatose', 'missing');
```

**Пример:**  
Narrative LLM: *«Барон Тёрн поднял кубок и произнёс тост за победу.»*  
Relational Canon: `Барон Тёрн.status = 'dead'` (убит в сцене 34).  
Результат: **CRITICAL** — мёртвый персонаж действует.

**Примечание:** Для `unconscious` и `imprisoned` проверка может быть более гибкой: `imprisoned` персонаж может действовать в сцене, которая происходит в тюрьме. Это решается дополнительным условием на локацию.

---

#### CHECK-D05: Нарушение временной последовательности (Timeline Violation)

**Описание:**  
Proposed Delta ссылается на событие, которое ещё не произошло по внутриигровому времени, или устанавливает время текущего события раньше, чем время последнего зафиксированного события.

**Severity:** `warning`

**SQL-логика:**

```sql
-- Проверка: новое событие не раньше последнего в той же сюжетной линии?
SELECT e.id, e.description, e.world_time, e.turn_id
FROM events e
WHERE e.campaign_id = :campaign_id
  AND e.world_time > :proposed_event_time
  AND e.story_thread_id = :story_thread_id
ORDER BY e.world_time DESC
LIMIT 5;
```

```sql
-- Проверка: ссылка на будущее событие?
SELECT e.id, e.description, e.world_time
FROM events e
WHERE e.id = :referenced_event_id
  AND e.campaign_id = :campaign_id
  AND e.world_time > :current_world_time;
```

**Пример:**  
Текущее мировое время: «3-й день, вечер».  
Narrative LLM: *«Лиара вспомнила вчерашний разговор с Сафирой на мосту.»*  
Relational Canon: встреча Лиары и Сафиры на мосту зафиксирована на «4-й день, утро».  
Результат: **WARNING** — Лиара вспоминает событие, которое ещё не произошло.

**Почему warning, а не critical:** Нарушение временной последовательности часто является результатом нечёткого трекинга мирового времени (NPC упоминает «вчера», а мировое время не обновилось). Это важный сигнал, но не всегда однозначная ошибка.

---

#### CHECK-D06: Knowledge Leak (Утечка знаний)

**Описание:**  
Персонаж в своей реплике или решении использует факт, который отсутствует в его `beliefs`. Это означает, что Narrative LLM наделила персонажа знанием, которого у него нет — нарушение модели субъективных знаний из Campaign Truth Engine (секция 5, product-foundation.md).

**Severity:** `critical`

**SQL-логика:**

```sql
-- Проверка: персонаж знает факт, на который ссылается?
-- Извлекаем все fact_ids, упомянутые в Proposed Delta для данного персонажа
-- и проверяем наличие соответствующих beliefs.

SELECT f.id AS fact_id, f.subject, f.predicate, f.object_value,
       f.truth_status
FROM facts f
WHERE f.id = :referenced_fact_id
  AND f.campaign_id = :campaign_id
  AND NOT EXISTS (
    SELECT 1 FROM beliefs b
    WHERE b.character_id = :char_id
      AND b.fact_id = f.id
      AND b.status IN ('known', 'believed', 'suspected')
      AND b.current_validity = TRUE
  );
```

**Пример:**  
Канон: Король жив и скрывается в монастыре (Fact #201).  
Beliefs: Лиара считает короля мёртвым (Belief: fact_id=201, status='believed', proposition='Король мёртв').  
Narrative LLM: *«Лиара сказала: 'Мы должны добраться до монастыря, пока Король в безопасности.'»*  
У Лиары нет belief с fact_id, связанным с локацией Короля в монастыре, и тем более с его статусом «в безопасности».  
Результат: **CRITICAL** — knowledge leak, персонаж знает то, чего не мог узнать.

**Примечание:** Это ключевая проверка для Campaign Truth Engine. Она требует, чтобы Context Compiler (секция 11.1) явно маркировал, какие `fact_ids` были переданы в контекст для каждого персонажа, а Proposed Delta содержал список `referenced_fact_ids` для каждой реплики NPC.

---

#### CHECK-D07: Дублирование предмета (Item Duplication)

**Описание:**  
После применения Proposed Delta один и тот же предмет оказывается одновременно в двух местах: у двух разных персонажей, или у персонажа и в локации, или в двух разных локациях.

**Severity:** `critical`

**SQL-логика:**

```sql
-- Проверка: предмет имеет более одного активного владельца/расположения?
-- Выполняется после виртуального применения delta к состоянию.
-- Используем CTE для моделирования нового состояния.

WITH projected_state AS (
  SELECT i.id, i.name,
         COALESCE(:new_owner_id, i.owner_id) AS owner_id,
         COALESCE(:new_location_id, i.location_id) AS location_id
  FROM items i
  WHERE i.id = :item_id AND i.campaign_id = :campaign_id
)
SELECT ps.id, ps.name
FROM projected_state ps
WHERE ps.owner_id IS NOT NULL 
  AND ps.location_id IS NOT NULL;

-- Дополнительно: проверка дублирования через несколько delta
SELECT item_id, COUNT(DISTINCT new_owner_id) AS owner_count
FROM proposed_delta_items
WHERE campaign_id = :campaign_id
GROUP BY item_id
HAVING COUNT(DISTINCT new_owner_id) > 1;
```

**Пример:**  
Proposed Delta содержит: `transfer_item(item=Карта Подземелья, to=Гарет)`.  
Но в том же ходе Narrative LLM описывает, что Лиара *«развернула Карту Подземелья на столе»*.  
Proposed Delta не содержит `remove_item(item=Карта Подземелья, from=Лиара)`.  
Результат: **CRITICAL** — предмет у двух персонажей одновременно.

#### CHECK-D08: Нарушение правил доступа к локации (Location Access Rule Violation)

**Описание:**  
Proposed Delta перемещает персонажа в локацию, к которой у него нет доступа по установленным правилам: закрытая дверь, охраняемый вход, магический барьер, или персонаж не принадлежит к фракции, имеющей доступ.

**Severity:** `warning`

**SQL-логика:**

```sql
-- Проверка: локация имеет правила доступа?
SELECT lar.id, lar.location_id, lar.rule_type, lar.condition,
       l.name AS location_name
FROM location_access_rules lar
JOIN locations l ON lar.location_id = l.id
WHERE lar.location_id = :target_location_id
  AND lar.campaign_id = :campaign_id
  AND lar.is_active = TRUE;

-- Проверка: персонаж удовлетворяет условиям доступа?
-- (конкретная логика зависит от rule_type)
-- Пример для faction-based доступа:
SELECT lar.id
FROM location_access_rules lar
WHERE lar.location_id = :target_location_id
  AND lar.rule_type = 'faction_membership'
  AND lar.campaign_id = :campaign_id
  AND lar.is_active = TRUE
  AND NOT EXISTS (
    SELECT 1 FROM character_factions cf
    WHERE cf.character_id = :char_id
      AND cf.faction_id = CAST(lar.condition AS INTEGER)
  );
```

**Пример:**  
Location Access Rule: «Королевская Сокровищница» — вход только для членов Королевской Гвардии или по личному разрешению Короля.  
Proposed Delta: `move_character(char=Гарет, to=Королевская Сокровищница)`.  
Гарет — торговец, не состоит в Королевской Гвардии.  
Результат: **WARNING** — нарушение правил доступа к локации.

**Почему warning, а не critical:** Правила доступа могут быть нарушены намеренно (вломился, прокрался, подкупил стражу). Это предупреждение для проверки, а не блокировка — возможно, Narrative LLM описывает именно взлом.

### 2.3. Общие характеристики Слоя 1

| Параметр | Значение |
|---|---|
| Стоимость LLM | 0 вызовов |
| Латентность | < 50 мс на все проверки суммарно |
| Precision | 100% — если SQL нашёл нарушение, оно реально |
| Recall | Ограничен: ловит только структурные нарушения |
| Блокирует ответ | Да, при severity = critical |
| Зависимость | Relational Canon, Proposed Delta |
| Порядок выполнения | После генерации, до отправки пользователю |

---

## 3. Слой 2 — LLM-проверки (нечёткие, асинхронные)

### 3.1. Принцип работы

Каждая проверка Слоя 2 — это отдельный LLM-вызов (или группа вызовов, объединённых в батч). На вход подаётся сгенерированный текст ответа, релевантный контекст из Relational Canon и профили персонажей. На выходе — структурированный JSON с вердиктом.

LLM-проверки запускаются **асинхронно, после** отправки ответа пользователю. Результаты отображаются в sidebar, Memory Inspector или как inline-предупреждения.

Для LLM-проверок можно (и рекомендуется) использовать **более дешёвую и быструю модель**, чем основная Narrative LLM. Задача проверки — аналитическая, не требует литературного стиля.

### 3.2. Перечень проверок

---

#### CHECK-L01: Нарушение характера (Out of Character — OOC)

**Описание:**  
Реакция, решение или реплика NPC не соответствует установленному профилю персонажа: характеру, ценностям, страхам, желаниям, текущему эмоциональному состоянию.

**Severity:** `warning`

**Формат промпта:**

```text
SYSTEM:
Ты — валидатор непрерывности персонажей в ролевой кампании. 
Твоя задача — оценить, соответствует ли поведение NPC его установленному 
профилю. Ответь в формате JSON.

USER:
## Профиль персонажа: {character_name}
- Характер: {character.personality}
- Ценности: {character.values}
- Страхи: {character.fears}
- Желания: {character.desires}
- Текущее эмоциональное состояние: {character.emotional_state}
- Текущие цели: {character.current_intentions}
- Речевые привычки: {character.speech_patterns}

## Контекст сцены
{scene_summary}
Последние 3 реплики:
{recent_turns}

## Оцениваемая реплика/действие
{generated_response_fragment}

## Вопрос
Соответствует ли данная реплика/действие установленному профилю персонажа? 
Учитывай характер, ценности, эмоциональное состояние и текущие цели.
```

**Input:**
- Профиль персонажа из Relational Canon (Character entity)
- Резюме текущей сцены
- Последние 3 хода
- Фрагмент сгенерированного ответа, относящийся к данному NPC

**Expected Output JSON:**

```json
{
  "check_id": "OOC_CHECK",
  "character_id": "char_456",
  "character_name": "Лиара",
  "is_consistent": false,
  "confidence": 0.85,
  "severity": "warning",
  "explanation": "Лиара описана как осторожная и подозрительная, но в данной реплике она безоговорочно доверяет незнакомцу и делится секретной информацией без колебаний. Это противоречит её установленным чертам 'подозрительность' и 'скрытность'.",
  "affected_traits": ["подозрительность", "скрытность"],
  "suggestion": "Лиара могла бы проверить незнакомца или потребовать доказательства, прежде чем раскрыть информацию."
}
```

**Estimated Cost:** ~300–800 input tokens + ~150–300 output tokens на одного персонажа. При использовании дешёвой модели (например, GPT-4o-mini, Gemini Flash, Llama 3 8B) — менее $0.001 за проверку.

---

#### CHECK-L02: Стилистика речи (Speech Pattern Drift)

**Описание:**  
Речь персонажа изменилась по стилю, лексике, синтаксису без обоснованной внутриигровой причины. Например, NPC, который всегда говорит короткими рублеными фразами, внезапно произносит длинный пафосный монолог. Или деревенский кузнец начинает использовать придворную лексику.

**Severity:** `info`

**Формат промпта:**

```text
SYSTEM:
Ты — лингвистический валидатор для ролевой кампании. 
Оцени, соответствует ли речь персонажа его установленному речевому паттерну.
Ответь в формате JSON.

USER:
## Речевой профиль: {character_name}
- Речевые привычки: {character.speech_patterns}
- Образование/социальный статус: {character.background}
- Примеры канонических реплик (из прошлых сцен):
  1. "{example_quote_1}"
  2. "{example_quote_2}"
  3. "{example_quote_3}"

## Текущая реплика
"{current_quote}"

## Контекст
Сцена: {scene_summary_brief}
Эмоциональное состояние: {character.emotional_state}

## Вопрос
Соответствует ли стиль речи установленному паттерну? 
Если нет, есть ли в контексте причина для изменения стиля 
(сильная эмоция, притворство, обращение к другой аудитории)?
```

**Input:**
- Речевой профиль персонажа
- 3 канонических примера реплик из прошлых сцен
- Текущая реплика
- Краткий контекст сцены

**Expected Output JSON:**

```json
{
  "check_id": "SPEECH_DRIFT",
  "character_id": "char_789",
  "character_name": "Страж",
  "is_consistent": true,
  "confidence": 0.72,
  "severity": "info",
  "explanation": "Речь немного более формальная, чем обычно, но Страж обращается к Королю, что оправдывает изменение регистра.",
  "justified_by_context": true
}
```

**Estimated Cost:** ~400–600 input tokens + ~100–200 output tokens. Менее $0.0005 за проверку.

#### CHECK-L03: Физическая правдоподобность (Physical Plausibility)

**Описание:**  
Описанное действие физически невозможно или крайне маловероятно с учётом установленного мира, физических законов (если не переопределены сеттингом) и возможностей персонажа.

**Severity:** `warning`

**Формат промпта:**

```text
SYSTEM:
Ты — валидатор физической правдоподобности для ролевой кампании.

Ответь в формате JSON.

USER:

- Мир: {campaign.setting_description}
- Магия: {campaign.magic_rules_summary}
- Технологический уровень: {campaign.tech_level}

## Действие
{action_description}

## Контекст
- Персонаж: {character_name}
- Способности: {character.abilities}
- Инвентарь: {character.inventory}
- Текущее состояние: {character.status}, {character.emotional_state}
- Локация: {location.description}
- Погодные условия: {scene.weather}

## Вопрос
Физически/логически возможно ли описанное действие в данном контексте? 
Учитывай законы физики (если не переопределены магией/сеттингом), 
способности персонажа и условия среды.
```

**Input:**
- Описание сеттинга и правил мира
- Описание действия из ответа Narrative LLM
- Профиль способностей персонажа
- Описание локации и условий

**Expected Output JSON:**

```json
{
```

#### CHECK-L04: Логическая непротиворечивость с каноном (Canon Consistency)

**Описание:**  
Сгенерированный текст содержит утверждения, факты или описания, которые противоречат установленному канону, но это противоречие — нечёткое, семантическое, не выразимое простым SQL-запросом. Например: «город процветает» vs. канонический факт «город разорён после осады»; «Лиара никогда не доверяла магам» vs. каноническое событие, где Лиара заключила союз с магом.

**Severity:** `warning`

**Формат промпта:**

```text
SYSTEM:
Ты — хранитель канона ролевой кампании. 
Проверь, не противоречит ли новый текст установленным каноническим фактам.
Ответь в формате JSON.

USER:
## Канонические факты (релевантные)
{relevant_facts_list}

## Канонические события (релевантные)
{relevant_events_list}

## Новый текст (ответ DM)
{generated_response}

## Вопрос
Содержит ли новый текст утверждения, описания или импликации, 
которые противоречат установленным каноническим фактам или событиям?
Обрати внимание на неявные противоречия (например, описание состояния 
локации, которое не соответствует последним событиям в ней).
```

**Input:**
- Список релевантных канонических фактов (извлечённых Context Compiler)
- Список релевантных канонических событий
- Полный текст сгенерированного ответа

**Expected Output JSON:**

```json
{
  "check_id": "CANON_CONSISTENCY",
  "is_consistent": false,
  "confidence": 0.88,
  "severity": "warning",
  "contradictions": [
    {
      "quote_from_response": "Рыночная площадь кипела жизнью, торговцы громко зазывали покупателей.",
      "conflicting_fact_id": "fact_892",
      "conflicting_fact": "Рыночная площадь Элдории разрушена после осады армией Теней (событие #45, сцена 28).",
      "explanation": "Площадь описана как функционирующая, но по канону она разрушена и не восстановлена."
    }
  ],
  "suggestion": "Описать площадь как разрушенную с признаками восстановления, или явно упомянуть, что прошло достаточно времени для восстановления."
}
```

**Estimated Cost:** ~500–1500 input tokens (зависит от количества фактов) + ~200–500 output tokens. ~$0.001–$0.003 за проверку.

#### CHECK-L05: Эмоциональная непрерывность (Emotional Continuity)

**Описание:**  
Эмоциональное состояние персонажа резко изменилось без достаточной внутриигровой причины. Например, NPC, который только что пережил гибель друга, через реплику весело шутит; или персонаж, находящийся в ярости, внезапно становится спокойным без де-эскалации.

**Severity:** `info`

**Формат промпта:**

```text
SYSTEM:
Ты — валидатор эмоциональной непрерывности для ролевой кампании.
Оцени, является ли эмоциональный переход персонажа правдоподобным.
Ответь в формате JSON.

USER:
## Персонаж: {character_name}
- Предыдущее эмоциональное состояние: {character.previous_emotional_state}
- Текущее эмоциональное состояние (по данным системы): {character.emotional_state}

## Последние события, затронувшие персонажа
{recent_events_affecting_character}

## Последние 3 реплики/действия персонажа
{recent_character_actions}

## Новая реплика/действие
{new_action_or_quote}

## Вопрос
Является ли эмоциональный тон новой реплики/действия естественным 
продолжением предыдущего эмоционального состояния персонажа? 
Если произошёл резкий переход, есть ли в контексте достаточная причина?
```

**Input:**
- Предыдущее и текущее эмоциональное состояние
- Последние события, затронувшие персонажа
- Последние 3 действия/реплики
- Новая реплика/действие

**Expected Output JSON:**

```json

  "check_id": "EMOTIONAL_CONTINUITY",
```

### 3.3. Общие характеристики Слоя 2

| Параметр | Значение |
|---|---|
| Стоимость LLM | 1 вызов на проверку (или 1–3 батчевых вызова на ход) |
| Латентность | 500–2000 мс на одну проверку |
| Precision | ~80–90% (возможны ложные срабатывания) |
| Recall | Высокий для семантических нарушений |
| Блокирует ответ | Нет |
| Зависимость | Relational Canon, Character Profiles, Generated Response |
| Порядок выполнения | Асинхронно, после отправки пользователю |

## 4. Когда каждый слой запускается

### 4.1. Слой 1: Синхронный, блокирующий

Слой 1 запускается **после генерации** ответа Narrative LLM и **до отправки** ответа пользователю. Это часть критического пути Turn Pipeline.

```text
[Narrative LLM генерирует ответ]
          │
          ▼
[Memory Scribe формирует Proposed Delta]
          │
          ▼
┌─────────────────────────────┐
│   СЛОЙ 1: SQL-проверки      │  ◄── СИНХРОННЫЙ, БЛОКИРУЮЩИЙ
│                             │
│   CHECK-D01..D08            │
│   Время: < 50 мс           │
│   Стоимость: 0 LLM          │
└─────────┬───────────────────┘
          │
          ├── Все пройдены ──► [Отправить ответ пользователю]
          │
          └── Critical найден ──► [Re-generate] (см. секцию 5)
```

**Поведение при critical:**
- Ответ **НЕ отправляется** пользователю.
- Запускается re-generation (максимум 2 попытки).
- Информация о нарушении передаётся в контекст для re-generation.

**Поведение при warning:**
- Ответ **отправляется** пользователю с пометкой ⚠️.
- Warning отображается в sidebar.

### 4.2. Слой 2: Асинхронный, неблокирующий

Слой 2 запускается **после отправки** ответа пользователю. Результаты появляются в sidebar/Memory Inspector по мере готовности.

```text
[Ответ отправлен пользователю]
          │
          ├──────────────────────────────┐
          │                              │
          ▼                              ▼
[Пользователь читает ответ]    ┌─────────────────────────┐
                               │  СЛОЙ 2: LLM-проверки   │  ◄── АСИНХРОННЫЙ
                               │                         │
                               │  CHECK-L01..L05         │
                               │  Время: 1-5 сек         │
                               │  Стоимость: 1-3 LLM     │
                               └────────┬────────────────┘
                                        │
                                        ▼
                               [Результаты в sidebar]
                               [Warnings: ⚠️ inline]
                               [Info: в Memory Inspector]
```

### 4.3. Конфигурация: режимы работы

Пользователь может выбрать один из трёх режимов через настройки кампании:

| Режим | Слой 1 | Слой 2 | Описание |
|---|---|---|---|
| **Relaxed** | Только critical (D01, D02, D04, D06, D07) | Отключен | Минимальные проверки, максимальная скорость. Для опытных пользователей, которые сами следят за каноном. |
| **Strict** (по умолчанию) | Все проверки (D01–D08) | Все проверки (L01–L05) | Полная валидация. Слой 1 блокирует при critical, Слой 2 предупреждает асинхронно. Баланс качества и скорости. |
| **Paranoid** | Все проверки (D01–D08) | Все проверки (L01–L05), **синхронно** | Оба слоя блокируют. Слой 2 запускается до отправки. Warning от L-проверок тоже требует подтверждения. Медленно, но максимальная гарантия канона. |

Дополнительные опции:

- `continuity_checker.layer2.model` — модель для LLM-проверок (может отличаться от Narrative LLM).
- `continuity_checker.layer2.batch_mode` — объединять несколько L-проверок в один LLM-вызов (экономия, но менее точно).
- `continuity_checker.max_retries` — максимальное количество re-generation при critical (по умолчанию: 2).
- `continuity_checker.ooc_cache_ttl` — время кеширования OOC-результата для персонажа (по умолчанию: 10 ходов).
- `continuity_checker.custom_checks_path` — путь к файлу с пользовательскими SQL-проверками.

## 5. Реакция на нарушение

### 5.1. Severity = Critical

Сценарий: Слой 1 обнаружил жёсткое структурное нарушение (knowledge leak, мёртвый NPC действует, предмет дублирован).

```text
[Critical нарушение обнаружено]
          │
          ▼
[НЕ показывать ответ пользователю]
          │
          ▼
[Попытка 1: Re-generate]
  - В контекст добавляется:
    "ВНИМАНИЕ: предыдущий ответ содержал нарушение канона:
     {violation_description}.
     Перегенерируй ответ, учитывая: {correction_instructions}."
          │
          ├── Проверки пройдены ──► [Отправить ответ]
          │
          └── Снова critical ──► [Попытка 2: Re-generate]
                                        │
                                        ├── Пройдено ──► [Отправить ответ]
                                        │
                                        └── Снова critical ──►
                                              [Отправить с warning ⚠️]
                                              [Пометка: "Автоматическая 
                                               валидация не пройдена"]
                                              [Логировать инцидент]
```

**Важно:** максимум 2 попытки re-generation. После этого ответ показывается с предупреждением — чтобы не блокировать игру бесконечно.

### 5.2. Severity = Warning

Ответ **отправляется** пользователю. Рядом с ответом (или в sidebar) отображается пометка:

```text
⚠️ Возможное нарушение: Нарушены правила доступа к локации 'Королевская 
Сокровищница'. Гарет не имеет права входа без разрешения.
   [Принять] [Отредактировать] [Перегенерировать]
```

Пользователь решает:
- **Принять** — warning снимается, ход фиксируется как есть. Если это осознанный выбор (персонаж вломился), можно добавить событие.
- **Отредактировать** — пользователь вручную редактирует текст ответа и/или Proposed Delta.
- **Перегенерировать** — запускается новая генерация с указанием на нарушение.

### 5.3. Severity = Info

Ответ отправляется без пометок в основном чате. Информация доступна в:
- **Memory Inspector** — раздел «Результаты валидации» для текущего хода.
- **Лог валидации** — полная история всех info-результатов.

Пример отображения в Memory Inspector:

```text
Ход #67 — Результаты валидации

✅ Слой 1: 8/8 проверок пройдены
⚠️ Слой 2: 4/5 проверок пройдены

ℹ️ CHECK-L05 (Эмоциональная непрерывность):
   Персонаж: Лиара
   Резкий переход от горя к веселью.
   Уверенность: 78%
   [Подробности] [Отклонить предупреждение]
```

### 5.4. Autofix: консервативное автоисправление

Для **некоторых** critical-нарушений система может применить автоматическое исправление состояния **без** re-generation текста.

Autofix применяется **только** когда:
- Исправление однозначно и безопасно.
- Исправление касается **состояния** (Relational Canon), а не **текста** ответа.
- Исправление **логируется** и доступно для review.

Примеры autofix:

| Нарушение | Autofix | Действие |
|---|---|---|
| Предмет в инвентаре мёртвого NPC | Переместить в локацию | `item.owner_id = NULL; item.location_id = npc.death_location_id;` |
| NPC действует в сцене, где его нет | Добавить в scene_participants | `INSERT INTO scene_participants (scene_id, character_id, joined_at)` |
| Дублирование предмета | Убрать из предыдущего владельца | `UPDATE items SET owner_id = new_owner WHERE id = item_id;` |

Каждый autofix генерирует запись:

```json
{
  "type": "autofix",
  "check_id": "ITEM_OWNERSHIP_CONFLICT",
  "action": "Предмет 'Кристалл Теней' автоматически убран из инвентаря 'Лиара' (мёртв) и перемещён в локацию 'Тёмный лес' (место гибели).",
  "turn_id": 67,
  "reversible": true,
  "timestamp": "2026-07-14T21:34:12Z"
}
```

**Принцип: conservative и logged.** Autofix никогда не изменяет текст нарратива и никогда не удаляет данные. Только перемещение/обновление статусов.

## 7. Стоимость и производительность

### 7.1. Слой 1: Нулевая стоимость LLM

| Метрика | Значение |
|---|---|
| LLM-вызовов на ход | 0 |
| Латентность (все 8 проверок) | < 50 мс |
| Стоимость в токенах | 0 |
| Стоимость в деньгах | $0.00 |
| Зависимость от сети | Нет (локальная БД) |

SQL-запросы к SQLite (или PostgreSQL) выполняются за единицы миллисекунд. Даже при 8 последовательных запросах суммарная латентность не превышает 50 мс. Это меньше, чем jitter сетевого запроса к LLM.

### 7.2. Слой 2: Управляемая стоимость LLM

| Метрика | Значение (без оптимизации) | Значение (с оптимизацией) |
|---|---|---|
| LLM-вызовов на ход | 5 (по одному на проверку) | 1–3 (батчинг) |
| Латентность (суммарная) | 2.5–10 сек | 1–5 сек |
| Input tokens на ход | ~2000–4500 | ~1500–3000 |
| Output tokens на ход | ~750–1550 | ~500–1000 |
| Стоимость (дешёвая модель) | ~$0.003–$0.006 | ~$0.001–$0.003 |
| Стоимость (основная модель) | ~$0.015–$0.035 | ~$0.008–$0.020 |

### 7.3. Стратегии оптимизации

#### Использование более дешёвой модели

Для LLM-проверок не нужна мощная модель, способная генерировать красивую прозу. Задача — **аналитическая**: сравнить текст с профилем и дать JSON-вердикт. Подходящие модели:

- **Облачные:** GPT-4o-mini, Gemini 2.0 Flash, Claude Haiku
- **Локальные:** Llama 3 8B, Mistral 7B, Qwen2.5 7B
- Стоимость: в 10–30 раз дешевле основной Narrative LLM

Конфигурация:

```yaml
continuity_checker:
  layer2:
    model: "gpt-4o-mini"  # или "ollama/llama3:8b" для локальной
    # отличается от narrative_model в основных настройках
```

#### Батчинг: несколько проверок в одном LLM-вызове

Вместо 5 отдельных вызовов — 1–3 комбинированных:

- **Батч 1:** CHECK-L01 (OOC) + CHECK-L02 (Speech) + CHECK-L05 (Emotional) — все связаны с профилем одного персонажа.
- **Батч 2:** CHECK-L03 (Physical) + CHECK-L04 (Canon) — связаны с контекстом мира.

Формат батчевого промпта:

```text
SYSTEM:
Проведи следующие проверки для ответа DM. 
Верни результат в формате JSON-массива.

USER:
## Проверки
1. OOC: соответствует ли поведение {char} профилю?
2. Speech: соответствует ли речь {char} паттерну?
3. Emotional: естественен ли эмоциональный переход?

## Данные
[профиль, контекст, реплика]

Ответь в формате:
[
  {"check_id": "OOC_CHECK", ...},
  {"check_id": "SPEECH_DRIFT", ...},
  {"check_id": "EMOTIONAL_CONTINUITY", ...}
]
```

**Экономия:** ~40–60% токенов за счёт общего контекста.

**Компромисс:** батчевая проверка может быть менее точной, так как модель обрабатывает несколько задач одновременно. Рекомендуется для режима **strict**, отдельные вызовы — для режима **paranoid**.

#### Кеширование результатов OOC-проверки

Профиль персонажа меняется редко (раз в несколько сцен). Если профиль не изменился и контекст сцены похож, результат OOC-проверки можно кешировать.

```python
cache_key = hash(
    character.personality_hash,    # Хеш профиля характера
    character.emotional_state,     # Текущее эмоциональное состояние
    scene.mood,                    # Настроение сцены
    last_ooc_check_turn_id         # Ход последней проверки
)
cache_ttl = 10  # ходов
```

Инвалидация кеша при:
- Изменении профиля персонажа (personality, values, fears).
- Значимом изменении эмоционального состояния.
- Смене сцены.
- Ручном запросе пользователя.

#### Избирательный запуск проверок

Не все проверки нужны на каждом ходу:

| Проверка | Условие запуска |
|---|---|
| CHECK-L01 (OOC) | Когда NPC совершает значимое действие или принимает решение |
| CHECK-L02 (Speech) | Когда NPC произносит реплику длиннее 2 предложений |
| CHECK-L03 (Physical) | Когда описано физическое действие (бой, перемещение, взаимодействие с объектами) |
| CHECK-L04 (Canon) | Всегда (но с кешированием контекста фактов) |
| CHECK-L05 (Emotional) | Когда обнаружено изменение эмоционального состояния в Proposed Delta |

### 7.4. Общий бюджет на ход

| Сценарий | Слой 1 | Слой 2 | Итого LLM | Итого время (добавка) | Итого стоимость |
|---|---|---|---|---|---|
| Relaxed | < 50 мс | — | 0 | < 50 мс | $0.00 |
| Strict (батч) | < 50 мс | 1–3 сек (async) | 1–2 вызова | < 50 мс (async не блокирует) | ~$0.001–$0.003 |
| Strict (без батча) | < 50 мс | 2.5–10 сек (async) | 3–5 вызовов | < 50 мс (async не блокирует) | ~$0.003–$0.006 |
| Paranoid (батч) | < 50 мс | 1–3 сек (sync) | 1–2 вызова | 1–3 сек | ~$0.001–$0.003 |
| Paranoid (без батча) | < 50 мс | 2.5–10 сек (sync) | 3–5 вызовов | 2.5–10 сек | ~$0.003–$0.006 |

## 9. Расширяемость

### 9.1. Плагинная система для SQL-проверок

Continuity Checker поддерживает пользовательские SQL-проверки, определённые через YAML или JSON конфигурацию. Это позволяет:

- Добавлять campaign-specific проверки (например, проверка магических ограничений конкретного сеттинга).
- Создавать ruleset-specific проверки (ограничения D&D, Pathfinder, GURPS).
- Распространять наборы проверок как плагины.

#### Формат пользовательской SQL-проверки (YAML):

```yaml
# custom_checks/magic_system.yaml

checks:
  - id: "CUSTOM_MANA_CHECK"
    name: "Проверка маны"
    description: "Персонаж не может использовать заклинание, если у него недостаточно маны."
    severity: "critical"
    enabled: true
    applies_to: "action:cast_spell"
    
    sql: |
      SELECT c.id, c.name, c.mana_current, s.mana_cost, s.name AS spell_name
      FROM characters c
      JOIN spells s ON s.id = :spell_id
      WHERE c.id = :char_id
        AND c.campaign_id = :campaign_id
        AND c.mana_current < s.mana_cost;
    
    message_template: >
      Персонаж '{char_name}' пытается использовать заклинание '{spell_name}' 
      (стоимость: {mana_cost} маны), но текущий уровень маны: {mana_current}.
    
    suggestion: "Уменьшить стоимость заклинания или описать провал каста."
    auto_fixable: false

  - id: "CUSTOM_DAYLIGHT_CHECK"
    name: "Проверка дневного света"
    description: "Вампир не может действовать при солнечном свете без защиты."
    severity: "warning"
    enabled: true
    applies_to: "character:has_trait:vampire"
    
    sql: |
      SELECT c.id, c.name, s.time_of_day, s.weather
      FROM characters c
      JOIN scenes s ON s.id = :scene_id
      JOIN character_traits ct ON ct.character_id = c.id
      WHERE c.id = :char_id
        AND ct.trait = 'vampire'
        AND s.time_of_day IN ('morning', 'noon', 'afternoon')
        AND s.weather NOT IN ('heavy_clouds', 'storm', 'eclipse')
        AND NOT EXISTS (
          SELECT 1 FROM character_items ci
          JOIN items i ON ci.item_id = i.id
          WHERE ci.character_id = c.id
            AND i.has_trait = 'sun_protection'
        );
    
    message_template: >
      Вампир '{char_name}' действует при солнечном свете ({time_of_day}) 
      без защиты от солнца.
    
    suggestion: "Добавить облачную погоду, укрытие или предмет защиты."
    auto_fixable: false
```

#### Формат пользовательской SQL-проверки (JSON):

```json
{
  "checks": [
    {
      "id": "CUSTOM_ENCUMBRANCE",
      "name": "Проверка перегруза",
      "description": "Персонаж не может нести больше предметов, чем позволяет его сила.",
      "severity": "warning",
      "enabled": true,
      "applies_to": "action:pick_up_item",
      "sql": "SELECT c.id, c.name, c.strength, COUNT(ci.item_id) AS item_count, (c.strength * 5) AS max_items FROM characters c LEFT JOIN character_items ci ON ci.character_id = c.id WHERE c.id = :char_id AND c.campaign_id = :campaign_id GROUP BY c.id HAVING COUNT(ci.item_id) >= (c.strength * 5)",
      "message_template": "Персонаж '{char_name}' (сила: {strength}) несёт {item_count} предметов при максимуме {max_items}.",
      "suggestion": "Оставить часть предметов или использовать транспорт.",
      "auto_fixable": false
    }
  ]
}
```

### 9.2. Campaign-level конфигурация

Каждая кампания может переопределить поведение Continuity Checker:

```yaml
# campaign_config.yaml

continuity_checker:
  mode: "strict"  # relaxed | strict | paranoid
  
  layer1:
    enabled: true
    checks:
      CHECK-D01: { enabled: true, severity: "critical" }
      CHECK-D02: { enabled: true, severity: "critical" }
      CHECK-D03: { enabled: true, severity: "critical" }
      CHECK-D04: { enabled: true, severity: "critical" }
      CHECK-D05: { enabled: true, severity: "warning" }
      CHECK-D06: { enabled: true, severity: "critical" }
      CHECK-D07: { enabled: true, severity: "critical" }
      CHECK-D08: { enabled: true, severity: "warning" }
    custom_checks_path: "./custom_checks/"
  
  layer2:
    enabled: true
    model: "gpt-4o-mini"
    batch_mode: true
    checks:
      CHECK-L01: { enabled: true, severity: "warning" }
      CHECK-L02: { enabled: true, severity: "info" }
      CHECK-L03: { enabled: true, severity: "warning" }
      CHECK-L04: { enabled: true, severity: "warning" }
      CHECK-L05: { enabled: true, severity: "info" }
    ooc_cache_ttl: 10  # ходов
  
  autofix:
    enabled: true
    allowed_fixes:
      - "ITEM_FROM_DEAD_NPC"
      - "ADD_TO_SCENE_PARTICIPANTS"
      - "REMOVE_DUPLICATE_ITEM"
    log_all: true
  
  max_retries: 2
```

### 9.3. Расширение LLM-проверок

Помимо встроенных L01–L05, пользователь или разработчик плагинов может добавить собственные LLM-проверки:

```yaml
# custom_checks/tone_check.yaml

llm_checks:
  - id: "CUSTOM_L_TONE"
    name: "Проверка тона повествования"
    description: "Тон повествования DM должен соответствовать установленному стилю кампании."
    severity: "info"
    enabled: true
    
    prompt_template: |
      SYSTEM:
      Оцени, соответствует ли тон текста установленному стилю кампании.
      
      USER:
      ## Стиль кампании
      {campaign.narrative_style}
      
      ## Текст ответа DM
      {generated_response}
      
      ## Вопрос
      Соответствует ли тон и стиль текста описанному стилю кампании?
    
    output_schema:
      type: "object"
      properties:
        is_consistent: { type: "boolean" }
        confidence: { type: "number" }
        explanation: { type: "string" }
    
    model_override: null  # использовать общую модель layer2
    run_condition: "always"  # always | on_scene_change | on_npc_action
```

### 9.4. Хуки и события

Continuity Checker генерирует события, на которые могут подписаться другие компоненты системы:

| Событие | Данные | Подписчики |
|---|---|---|
| `validation.layer1.completed` | Список результатов всех D-проверок | Turn Pipeline, Logger |
| `validation.layer2.completed` | Список результатов всех L-проверок | UI (sidebar), Memory Inspector |
| `validation.critical_found` | Детали нарушения | Re-generation Loop, Logger |
| `validation.autofix_applied` | Детали автоисправления | Logger, Memory Inspector, UI |
| `validation.retries_exhausted` | Детали после max retries | UI (warning badge), Logger |

## Приложение A: Глоссарий

| Термин | Определение |
|---|---|
| **Relational Canon** | Основное хранилище структурированного состояния кампании (SQLite/PostgreSQL). Содержит entities, facts, beliefs, events, scenes. |
| **Proposed Delta** | Структурированный набор изменений, предложенный Memory Scribe на основе ответа Narrative LLM. Применяется к Relational Canon только после валидации. |
| **Knowledge Scope** | Набор фактов и beliefs, доступных конкретному персонажу. Определяется Context Compiler на основе presence, beliefs, events. |
| **Autofix** | Автоматическое консервативное исправление состояния Relational Canon при обнаружении critical-нарушения. Не затрагивает текст ответа. |
| **Turn Pipeline** | Полный цикл обработки одного хода: от ввода пользователя до сохранения результата. |
| **Re-generation** | Повторная генерация ответа Narrative LLM с дополнительными инструкциями о выявленном нарушении. |
