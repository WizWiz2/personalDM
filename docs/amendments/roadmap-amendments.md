## 2. Этап 0. Vertical Skeleton (расширенный)

### 2.1. Оригинальный список (из §19)

- создать кампанию
- подключить local/OpenAI-compatible LLM
- streaming chat
- сохранить каждый ход
- перезапустить приложение
- продолжить диалог
- экспортировать лог

### 2.2. Добавления

К оригинальному списку **обязательно** добавить:

#### Sliding Window для истории ходов

**Что:** базовый context management — последние N ходов попадают в prompt.

**Почему на этапе 0:** без sliding window уже на 20-м ходе prompt переполнит context window. Система не сможет вести даже короткий диалог.

**Минимальная реализация:**
- Turn-count based (последние N ходов, например 20)
- Без token counting (это Этап 1)
- Без суммаризации выпавших ходов (это Этап 2+)

#### Базовый System Prompt с описанием кампании

**Что:** system prompt, включающий название кампании, описание мира, стиль повествования.

**Почему на этапе 0:** без system prompt LLM не знает, что она ведёт RPG. Ответы будут бессмысленны.

**Минимальная реализация:**
- Шаблон system prompt с placeholders: `{campaign_name}`, `{campaign_description}`, `{style}`
- Пользователь заполняет при создании кампании
- System prompt отправляется первым сообщением в каждом запросе к LLM

#### Прерывание и продолжение streaming

**Что:** возможность остановить генерацию ответа DM на середине и получить частичный результат.

**Почему на этапе 0:** LLM может генерировать очень длинный ответ. Без кнопки «Стоп» пользователь ждёт бесконечно.

**Минимальная реализация:**
- Кнопка «Stop» в UI во время streaming
- При нажатии: abort запрос к LLM
- Сохранить полученную часть ответа как полный Turn
- State delta от частичного ответа: не применять (или применять с пометкой `partial`)

**Edge cases:**
- Что если ответ оборван посередине предложения? → Сохранить как есть, пометить `truncated: true`
- Можно ли «продолжить» оборванный ответ? → На этапе 0 — нет. На этапе 1+ — через regenerate или continue

#### Обработка ошибок LLM

**Что:** graceful handling ситуаций, когда LLM не может ответить.

**Почему на этапе 0:** без error handling первая ошибка LLM ломает весь UI.

**Сценарии и обработка:**

| Ошибка | Обработка |
|--------|-----------|
| **Таймаут** (LLM не ответила за N секунд) | Уведомление «Модель не ответила. Повторить?». Настраиваемый timeout (30s-300s) |
| **Connection refused** (Ollama не запущен) | «Не удалось подключиться к модели. Проверьте, что Ollama запущен на {url}» |
| **Rate limit** (облачный API) | Автоматический retry с exponential backoff. Показать countdown. |
| **Invalid response** (не JSON, обрезанный ответ) | Retry до 3 раз. Если не помогает — показать raw ответ + предложить regenerate |
| **Content filter** (облачный API отклонил) | «Модель отклонила запрос (content policy). Попробуйте переформулировать ход или сменить модель.» |
| **Model not found** | «Модель {name} не найдена. Доступные модели: {list}» |
| **Context overflow** | «Контекст превышает лимит модели. Уменьшаем окно истории.» Автоматически reduce sliding window |

#### Простая оценка длины контекста

**Что:** примерная оценка размера prompt перед отправкой.

**Почему на этапе 0:** без оценки легко превысить context window модели.

**Минимальная реализация:**
- Приблизительный подсчёт: `len(text) / 4` для английского, `len(text) / 2.5` для русского
- Сравнить с `model.context_length`
- Если > 90% — урезать sliding window
- Показывать в UI: «Контекст: ~3200/4096 tokens»

Точный tokenizer не нужен на этапе 0. Достаточно safe approximation с запасом.

#### Простой Undo (откат последнего хода)

**Что:** кнопка «Отменить» для удаления последнего хода (user + DM response).

**Почему на этапе 0:** без undo пользователь не может исправить свой ввод или отклонить плохой ответ DM. Это базовый UX.

**Минимальная реализация:**
- Soft-delete последнего Turn
- Без state delta revert (на этапе 0 нет structured state)
- Запись в raw archive: «undo of turn N»

### 2.3. Итоговый чеклист Этапа 0


### 2.4. Критерий завершения Этапа 0

5. Откатить последний ход
6. Закрыть приложение и продолжить
7. Экспортировать лог

## 3. НОВЫЙ Этап 1.5. Assisted Canon

### 3.1. Обоснование

Между Этапом 1 (Manual Canon) и Этапом 2 (Memory Scribe) — большой gap:

- **Этап 1:** пользователь *вручную* создаёт всех персонажей, все факты, все отношения.
- **Этап 2:** система *автоматически* извлекает structured data из каждого хода, с полным FSM, blacklist, conflict detection.

Этап 1.5 — промежуточный: система *предлагает*, пользователь *подтверждает*. Это проще, чем полный Memory Scribe, но уже снимает основную боль ручного ведения канона.

### 3.2. Scope Этапа 1.5

#### LLM предлагает факты/события после каждого хода

После каждого ответа DM, отдельный вызов LLM (или тот же, с structured output) генерирует:
- Новые факты («Лиара получила письмо от Сафиры»)
- Новые события («Встреча в тронном зале»)
- Изменения отношений («Доверие Лиары к стражу снизилось»)
- Перемещения персонажей («Лиара перешла в библиотеку»)
- Обновления сцены (настроение, напряжение)

**Не входит в 1.5** (отложено до Этапа 2):
- Beliefs и subjective knowledge (кто что знает/подозревает)
- Blacklist и семантическая фильтрация
- Иерархическое разрешение конфликтов
- Auto-approve
- Batch review

#### Пользователь подтверждает через простой UI

Карточки появляются после каждого хода DM:

```
┌─────────────────────────────────────────────────┐
│ 💡 Memory Scribe предлагает записать:           │
│                                                  │
│ ┌────────────────────────────────────────────┐   │
│ │ 📝 Факт: "Лиара получила запечатанное      │   │
│ │   письмо от Сафиры"                        │   │
│ │                                             │   │
│ │ [✓ Принять]  [✗ Отклонить]  [✏️ Править]   │   │
│ └────────────────────────────────────────────┘   │
│                                                  │
│ ┌────────────────────────────────────────────┐   │
│ │ 🔗 Отношение: Лиара → Сафира              │   │
│ │   Подозрение: 0.2 → 0.4                    │   │
│ │                                             │   │
│ │ [✓ Принять]  [✗ Отклонить]  [✏️ Править]   │   │
│ └────────────────────────────────────────────┘   │
│                                                  │
│ [✓ Принять все]              [Пропустить все]   │
└─────────────────────────────────────────────────┘
```

#### Метрика успеха

**Процент автоматически одобренных предложений:**
- Если пользователь одобряет >80% предложений без правок → extraction работает хорошо
- Если <50% → нужно улучшить prompt для extraction или сузить scope
- Собирать статистику для информирования решений по Этапу 2 (auto-approve thresholds)

#### Опциональный auto-approve с порогом confidence

В настройках кампании (для продвинутых):
- «Автоматически принимать предложения с уверенностью выше X%»
- По умолчанию: выключено
- Рекомендуемый диапазон: 85-95%

Это подготовка к полному auto-approve в Этапе 2.

### 3.3. Минимальный FSM для 1.5

Упрощённый по сравнению с полным FSM из memory-pipeline.md:

```
proposed → approved → active
proposed → rejected (без blacklist)
proposed → edited → active
```

Полный FSM (disputed, blacklisted, superseded, pending_review) — Этап 2.

### 3.4. Критерий завершения Этапа 1.5

1. После каждого хода DM появляются 1-5 предложений (факты, события, отношения)
2. Пользователь может принять/отклонить/отредактировать каждое
3. Принятые предложения становятся active и попадают в контекст LLM
4. Отклонённые исчезают (без blacklist)
5. Статистика: процент одобрений видна в Settings

## 4. Этап 7: разделение Rules Engine и Multiplayer

### 4.1. Проблема текущей формулировки

Этап 7 в product-foundation.md объединяет две принципиально разные задачи:

```
Текущий Этап 7: Rules and Multiplayer
- dice                          ← Rules Engine
- character sheets              ← Rules Engine
- inventory                     ← Rules Engine
- pluggable rulesets            ← Rules Engine
- deterministic combat          ← Rules Engine
- private player knowledge      ← Multiplayer
- multiplayer                   ← Multiplayer
```

**Почему это проблема:**

Multiplayer — это не «ещё одна фича». Это фундаментальное изменение архитектуры:

| Аспект | Single-player | Multiplayer |
|--------|--------------|-------------|
| Auth | Не нужен | OAuth/sessions, роли (DM, Player) |
| State management | Один пользователь, один state | Concurrent access, race conditions |
| Privacy | Всё видно | Private knowledge per player, hidden DM notes |
| Sync | Не нужен | Real-time sync (WebSocket), conflict resolution |
| Branching | Один пользователь решает | Кто может ветвить? Голосование? |
| Memory Scribe | Один reviewer | Кто approve'ит? DM? Все? |
| Turn order | Свободный | Инициатива, очередь, параллельные действия |
| Scaling | SQLite на одном устройстве | PostgreSQL, возможно Redis для pub/sub |
| Deployment | Desktop app | Server + multiple clients |
| Data model | `user_id` не нужен | Каждая сущность привязана к owner/visibility |

Добавление multiplayer затрагивает *каждый* слой приложения. Это не расширение — это параллельный продукт.

### 4.2. Предложение: разделить Этап 7

**Новый Этап 7: Rules Engine** (single-player)
- Dice roller с поддержкой формул (2d6+3, d20 advantage, etc.)
- Character sheets (HTML/JSON templates, привязанные к ruleset)
- Inventory management (вес, слоты, equipped/stored)
- Pluggable rulesets:
  - Systemless (свободная форма, нет правил)
  - D&D 5e (или SRD-compatible subset)
  - Простая d6-система
  - Custom JSON-based ruleset
- Deterministic combat (инициатива, HP, AC, damage calculations)
- LLM использует dice через tool calls, не «выдумывает» результаты

**Новая Фаза 2: Multiplayer** (отдельный product foundation)
- Собственный product-foundation-multiplayer.md
- Собственная архитектура (server mode, PostgreSQL, WebSocket)
- Собственная модель данных (user roles, visibility, permissions)
- Собственный roadmap (может начинаться параллельно с поздними этапами Фазы 1)

### 4.3. Обоснование выноса

1. **Фокус:** работа над Rules Engine не требует переосмысления архитектуры. Multiplayer — требует.
2. **Time-to-value:** Rules Engine нужен многим single-player пользователям. Multiplayer — нишевая потребность на раннем этапе.
3. **Risk:** если Multiplayer провалится, Rules Engine не пострадает.
4. **Dependencies:** Rules Engine зависит только от core (Этапы 0-2). Multiplayer зависит от *всего*.

## 7. Рекомендации по приоритизации

### 7.1. Если один разработчик

Строго последовательно по критическому пути: 0 → 1 → 1.5 → 2 → 3 → 4.

Visual и Music — после Этапа 4, когда core стабилен.

### 7.2. Если два разработчика

- **Dev 1 (Backend):** 0 → 1 → 1.5 → 2 → 3 → 4
- **Dev 2 (Frontend + Media):** UI для 0 → UI для 1 → Visual Layer (Этап 5) → Music Layer (Этап 6) → UI для 3-4

### 7.3. Что можно срезать для faster MVP

Если нужно показать продукт раньше:

| Срезать | Эффект | Последствие |
|---------|--------|-------------|
| Этап 1.5 (Assisted Canon) | -3 нед | Больше ручной работы для пользователя, но Manual Canon работает |
| Continuity Engine (Этап 4) | -6 нед | Нет автоматических проверок, но Memory Inspector позволяет ручной контроль |
| Rules Engine (Этап 7) | Нет потерь для MVP | Systemless mode — допустимый MVP |

**Минимальный playable MVP: Этапы 0 + 1 + 2 + 3** (~17 недель, ~4 месяца).

Это даёт: streaming chat + structured canon + автоматическая извлечение + semantic search. Enough to wow.

## 8. Risk Mitigation по этапам

| Этап | Главный риск | Mitigation |
|------|-------------|------------|
| 0 | LLM API нестабилен/медленен | Retry logic, timeout config, mock mode для разработки |
| 1 | Модель данных слишком сложная или слишком простая | Начать с minimum viable schema (Entity, Fact, Relationship). JSONB для экспериментальных полей |
| 1.5 | LLM плохо извлекает structured data | Несколько prompt templates. Fallback на простой regex extraction. Метрика approval rate |
| 2 | Complexity explosion в FSM | Реализовать минимальный FSM (proposed→active/rejected). Добавлять states по мере необходимости |
| 3 | Vector search даёт нерелевантные результаты | Hybrid approach (vector + FTS + entity overlap). Reranker. Fallback на FTS-only |
| 4 | Ложные срабатывания continuity checks | Soft warnings, а не hard blocks. Пользователь решает |
| 5 | ComfyUI сложна в интеграции | Начать с простого txt2img. Workflow усложнять позднее |
| 7 | Абстракция rulesets слишком сложна | Начать с одной системы (или systemless). Plug-in architecture позже |
