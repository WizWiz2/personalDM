## Преамбула

Context Compiler — самый критичный компонент всей системы «Личный ДМ».

LLM не имеет прямого доступа к базе данных. Она не видит Campaign Truth Engine, не читает Хроники Акаши напрямую, не знает о существовании тысяч фактов, событий и убеждений, хранимых в structured storage. **LLM видит только и исключительно то, что Context Compiler поместил в её context window.** Всё остальное для неё не существует.

Если Context Compiler плохо подобрал контекст:

- NPC забудет события, свидетелем которых он был;
- NPC узнает секреты, которых не мог знать;
- сцена потеряет атмосферу, потому что описание среды не попало в промпт;
- сюжетные линии оборвутся, потому что модель не увидела их статус;
- игрок получит ответ, противоречащий канону, потому что канон не был включён.

Если Context Compiler хорошо подобрал контекст:

- NPC ведёт себя натурально, руководствуясь своими знаниями и целями;
- атмосфера сцены остаётся цельной;
- модель вспоминает старые события через retrieval;
- continuity violations минимальны;
- пользователю кажется, что ДМ действительно «помнит» кампанию.

Context Compiler — это линза, через которую бесконечная реальность кампании проецируется в конечное окно восприятия модели. Качество этой линзы определяет качество всего продукта.

> Context Compiler не генерирует контент. Он формирует *реальность*, в которой LLM существует в момент генерации ответа.

## 1. Бюджет токенов

### 1.1. Фиксированный vs адаптивный бюджет

Context Compiler должен поддерживать два режима работы с бюджетом:

**Адаптивный режим (по умолчанию):**  
Бюджет автоматически рассчитывается на основе context window подключённой модели. При смене модели (или provider'а) бюджет пересчитывается автоматически.

**Фиксированный режим (ручной override):**  
Пользователь может задать жёсткий лимит вручную. Это полезно для:

- тестирования поведения системы при ограниченном контексте;
- экономии при платных API (меньше input-токенов — меньше расходов);
- A/B-тестирования: сравнение качества ответов при разных бюджетах;
- принудительного ограничения для слабых моделей, которые деградируют на длинных контекстах.

### 1.2. Формула расчёта

```
model_context_window      = <из конфигурации provider'а, токены>
reserved_for_response     = <токены, зарезервированные для ответа модели>
safety_margin             = model_context_window × safety_margin_percent
max_context_budget        = model_context_window - reserved_for_response - safety_margin
```

**Параметры конфигурации:**

| Параметр | Описание | Значение по умолчанию | Источник |
|----------|----------|-----------------------|----------|
| `model_context_window` | Размер context window модели в токенах | Зависит от модели | Provider config / model metadata |
| `reserved_for_response` | Токены, зарезервированные для ответа модели (`max_tokens` в API-вызове) | 2048 | Конфигурация кампании |
| `safety_margin_percent` | Запас безопасности в % от context window | 5% | Конфигурация системы |
| `manual_budget_override` | Ручной override максимального бюджета (если задан, игнорирует расчёт) | `null` | Конфигурация кампании |

**Пример расчёта для модели с 32k context window:**

```
model_context_window    = 32768
reserved_for_response   = 2048
safety_margin           = 32768 × 0.05 = 1638
max_context_budget      = 32768 - 2048 - 1638 = 29082 токенов
```

**Пример для модели с 128k context window:**

```
model_context_window    = 131072
reserved_for_response   = 4096
safety_margin           = 131072 × 0.05 = 6554
max_context_budget      = 131072 - 4096 - 6554 = 120422 токенов
```

### 1.3. Safety margin

Safety margin (5–10% от context window) необходим по следующим причинам:

- **Погрешность подсчёта токенов.** Разные tokenizer'ы дают разные числа. При использовании fallback-метода (`chars/4`) погрешность может достигать ±15%.
- **Динамические дополнения.** Некоторые provider'ы добавляют системные токены (special tokens, chat template markers), которые не учитываются при сборке контекста.
- **Защита от обрезки.** При превышении context window модель молча обрезает начало контекста. Это катастрофично, потому что system prompt находится в начале.
- **Overhead формата.** JSON/XML-разметка, маркеры ролей (`user:`, `assistant:`), разделители сообщений занимают токены сверх полезного содержимого.

Рекомендация:
- 5% для provider'ов с точным tokenizer'ом (OpenAI, Anthropic);
- 7–10% для локальных моделей, где tokenizer может не полностью совпадать.

### 1.4. Подсчёт токенов

Context Compiler должен подсчитывать токены для каждого блока контекста перед включением. Поддерживаются два метода:

**Точный подсчёт (tiktoken):**

```python
import tiktoken

def count_tokens_tiktoken(text: str, model: str = "gpt-4") -> int:
    encoding = tiktoken.encoding_for_model(model)
    return len(encoding.encode(text))
```

Используется для OpenAI-совместимых моделей, где tokenizer известен. Если provider явно указывает encoding (например, `cl100k_base`, `o200k_base`), используется он.

**Приблизительный подсчёт (fallback):**

```python
def count_tokens_approx(text: str) -> int:
    return max(1, len(text) // 4)
```

Используется когда:
- модель запущена через Ollama/KoboldCpp без указания tokenizer'а;
- encoding неизвестен;
- нужна быстрая предварительная оценка.

**Приоритет выбора метода:**

1. Если provider сообщает `encoding_name` → `tiktoken` с указанным encoding.
2. Если provider сообщает `model_name` и он есть в реестре tiktoken → `tiktoken` с определением encoding по модели.
3. Если provider предоставляет собственный tokenizer endpoint → вызов API (кеширование обязательно).
4. Fallback → `chars / 4`.

**Кеширование:**

Подсчёт токенов для неизменных блоков (system prompt, campaign instructions, NPC knowledge blocks) кешируется. Пересчёт происходит только при изменении содержимого блока. Кеш-ключ — хеш содержимого блока.

### 1.5. Конфигурация бюджета ответа

Параметр `reserved_for_response` определяет, сколько токенов остаётся модели для генерации ответа. Это значение одновременно:

- вычитается из budget'а контекста;
- передаётся как `max_tokens` (или аналог) в API-вызов.

Рекомендуемые значения:

| Сценарий | `reserved_for_response` |
|----------|------------------------|
| Стандартный нарративный ход | 1024–2048 |
| Длинное описание сцены | 2048–4096 |
| Краткий ответ NPC | 512 |
| Structured extraction (Scribe) | 1024 |
| Боевой раунд | 1024 |

Значение может динамически изменяться Turn Runner'ом в зависимости от контекста хода, но Context Compiler использует фиксированное значение из конфигурации как baseline.

## 2. Приоритизация слоёв

Context Compiler организует контекст в иерархию слоёв с убывающим приоритетом. Каждый слой имеет чётко определённое содержимое, примерный размер и стратегию компрессии при нехватке бюджета.

### Слой 0: System Prompt + Campaign Instructions

**Приоритет:** ОБЯЗАТЕЛЬНЫЙ. Включается ВСЕГДА. Не подлежит сокращению.

**Содержимое:**

- Базовый system prompt: роль модели как ДМа, правила поведения, ограничения.
- Campaign-specific instructions: стиль повествования, сеттинг, тон, ограничения жанра.
- Tool descriptions: описание доступных tool calls (если используются).
- Output format instructions: формат ожидаемого ответа (для scribe/extraction задач).
- Глобальные ограничения: запрет на всеведение NPC, правила работы с каноном, правила конфиденциальности знаний.

**Примерный размер:** 500–2000 токенов.

**Стратегия компрессии:** Не сжимается. Если system prompt не помещается в бюджет, это ошибка конфигурации, о которой система предупреждает пользователя.

**Формат:**

```
[SYSTEM]
Ты — мастер подземелий (ДМ) в продолжительной кампании "{campaign.name}".
{campaign.system_instructions}

Правила:
- Ты НЕ знаешь того, чего не знают присутствующие NPC.
- Ты НЕ изменяешь установленные факты без причины.
- Ты используешь tool calls для бросков кубиков и изменений состояния.
{campaign.style_guidelines}

{tool_descriptions}
{output_format}
```

### Слой 1: Текущая сцена

**Приоритет:** ОБЯЗАТЕЛЬНЫЙ. Включается ВСЕГДА. Минимальная версия обязательна.

**Содержимое:**

- Название и описание текущей локации.
- Время суток, погода, освещение.
- Список присутствующих персонажей (имя, краткое состояние, позиция в сцене).
- Текущее настроение / атмосфера / напряжение.
- Unresolved beats: незавершённые действия, повисшие вопросы, ожидаемые реакции.
- Активные эффекты: заклинания, состояния, таймеры.
- Краткое summary предыдущей сцены (для контекста перехода).

**Примерный размер:** 300–1500 токенов.

Зависит от сложности сцены. Таверна с двумя NPC — ~300 токенов. Тронный зал с 8 NPC, ритуалом и политическими интригами — ~1500 токенов.

**Стратегия компрессии:**

- Полное описание при достаточном бюджете.
- При дефиците бюджета: сжатое описание локации (1–2 предложения), только имена присутствующих без состояний, только критические unresolved beats.
- Минимально: название локации + список имён присутствующих + 1 главный unresolved beat.

**Формат:**

```
[ТЕКУЩАЯ СЦЕНА]
Локация: Тронный зал замка Валтергард
Время: Поздний вечер, свечи и жаровни.
Атмосфера: Напряжённое ожидание. Совет военачальников перед битвой.

Присутствуют:
- Лорд Кейран (стоит у карты, раздражён)
- Лиара (сидит у окна, следит за советниками)
- Капитан Ренн (при оружии, у двери)

Незавершённые линии:
- Лорд Кейран задал вопрос о судьбе пленного — ожидает ответа игрока.
- Лиара хочет поговорить наедине — ждёт момента.
```

### Слой 2: Знания и цели присутствующих NPC

**Приоритет:** ВЫСОКИЙ. Включается после обязательных слоёв. Критически влияет на качество roleplay.

**Содержимое:**

Для каждого NPC, присутствующего в текущей сцене, формируется structured блок, содержащий:

- Краткое описание характера и текущего эмоционального состояния.
- Текущие цели (в контексте сцены и долгосрочные).
- Ключевые знания: что NPC знает, что считает истинным (beliefs), что подозревает.
- Ключевые отношения: отношение к игроку, к другим присутствующим NPC.
- Секреты: что NPC скрывает от конкретных лиц.
- Ограничения: что NPC НЕ знает (явный negative context для предотвращения утечек).

**Примерный размер:** 200–600 токенов на NPC. При 3–5 NPC: 600–3000 токенов.

**Стратегия компрессии:**

1. **Полный блок** (≥400 токенов на NPC): все знания, все отношения, все цели.
2. **Сжатый блок** (~200 токенов на NPC): только текущая цель, ключевые знания, отношение к игроку.
3. **Минимальный блок** (~50 токенов на NPC): имя, одно предложение о намерении, одно ключевое знание.
4. **Исключение NPC**: наименее релевантные NPC удаляются полностью (см. секцию Truncation).

Приоритизация NPC внутри слоя:
1. NPC, с которым сейчас идёт прямой диалог.
2. NPC, упомянутый в последнем ходе игрока.
3. NPC с активным unresolved beat.
4. NPC с высоким importance score.
5. NPC, просто присутствующий фоном.

**Подробнее о формате — см. секцию 4 (Формат инъекции знаний NPC).**

### Слой 3: Последние ходы (Sliding Window)

**Приоритет:** ВЫСОКИЙ. Определяет способность модели поддерживать связность разговора.

**Содержимое:**

Последние N ходов диалога между пользователем и ДМом. Каждый ход включает:

- Роль (user / assistant / system).
- Текст хода.
- Tool calls и их результаты (если были).
- Результаты механик (броски, проверки).

**Примерный размер:** 100–500 токенов на ход. При 10–20 ходах: 1000–10000 токенов.

**Стратегия заполнения:**

Sliding window заполняется **от новых ходов к старым**:

1. Всегда включать минимум **3 последних хода** (user-assistant пары). Это абсолютный минимум для поддержания связности.
2. Добавлять ходы по одному, начиная от самого нового, пока позволяет бюджет.
3. Если ход содержит tool calls, включать и вызов, и результат — это одна неделимая единица.
4. Максимальная глубина: ограничена конфигурацией (`max_turn_window_depth`, по умолчанию 50).

**Стратегия компрессии старых ходов:**

- Ходы, попадающие в «хвост» window, могут быть заменены на сжатый summary:  
  `[Ходы 15-25: Игрок исследовал подземелье, встретил ловушку, нашёл свиток.]`
- Это позволяет сохранить temporal context без полной стоимости токенов.
- Summary генерируется Memory Scribe и хранится в базе.

**Формат:**

```
[ИСТОРИЯ ДИАЛОГА]

[Ход 42 — Игрок]
Я подхожу к Лиаре и шёпотом спрашиваю: «Ты знаешь, что случилось с королём?»

[Ход 42 — ДМ]
Лиара бросает быстрый взгляд на Лорда Кейрана, убеждаясь, что он не слышит...
{продолжение ответа}

[Ход 43 — Игрок]
{текст}
```

### Слой 4: Retrieval Results

**Приоритет:** СРЕДНИЙ. Обеспечивает долговременную память — способность вспоминать события из далёкого прошлого кампании.

**Содержимое:**

Результаты hybrid retrieval, запущенного на основе текущего контекста хода:

- Semantic search: результаты vector similarity по embeddings текущего хода и сцены.
- Keyword search: FTS5 результаты по именам, локациям, ключевым словам.
- Entity overlap: записи, связанные с теми же сущностями, что и текущая сцена.

Каждый результат представляет собой Memory Chunk (см. product-foundation.md, секция 10) и содержит:

- Текст фрагмента.
- Relevance score (нормализованный, 0.0–1.0).
- Source: откуда пришёл фрагмент (событие, ход, факт, summary).
- Temporal marker: когда это произошло в игровом времени.
- Entity links: связанные сущности.

**Примерный размер:** 100–300 токенов на chunk. Обычно 3–10 chunks: 300–3000 токенов.

**Стратегия заполнения:**

1. Retrieval pipeline возвращает ранжированный список кандидатов.
2. Кандидаты включаются в порядке убывания relevance score.
3. Минимальный порог relevance: `min_relevance_threshold` (по умолчанию 0.3).
4. Дедупликация: если два chunk'а покрывают одно и то же событие, включается тот, чей score выше.
5. Заполнение останавливается при исчерпании бюджета.

**Обязательные фильтры (применяются ДО ранжирования):**

- `campaign_id`, `branch_id` — изоляция данных кампании и ветки.
- `visibility` — только записи, видимые в текущем контексте.
- `valid_interval` — только актуальные записи (не superseded).
- `character_knowledge_scope` — для NPC-специфичных запросов: только то, что NPC мог знать.
- `deleted/superseded` — исключение удалённых и заменённых записей.

**Формат:**

```
[РЕЛЕВАНТНЫЕ ВОСПОМИНАНИЯ]

Ход 12, 3 дня назад (внутриигровое время):
Лиара получила письмо от Сафиры, в котором сообщалось о смерти короля.
Лиара была потрясена и приняла известие за правду.
(relevance: 0.91)

Ход 28, 1 день назад:
Стражник на воротах монастыря отказал группе в проходе, ссылаясь на карантин.
Монастырь закрыт уже неделю.
(relevance: 0.72)
```

### Слой 5: Активные сюжетные линии (Story Threads)

**Приоритет:** СРЕДНИЙ-НИЗКИЙ. Помогает модели поддерживать долгосрочные арки.

**Содержимое:**

Список активных Story Threads (см. product-foundation.md, секция 10) с текущим статусом:

- Формулировка: о чём эта сюжетная линия.
- Статус: active / suspended / near_resolution.
- Stakes: что поставлено на кон.
- Открытые вопросы: что ещё не разрешено.
- Последнее событие по этой линии.
- Участники.

**Примерный размер:** 50–200 токенов на thread. При 3–7 активных threads: 150–1400 токенов.

**Стратегия компрессии:**

1. **Полный блок**: все поля для каждого thread.
2. **Сжатый блок**: только формулировка + статус + один открытый вопрос.
3. **Минимальный**: только формулировка в одну строку.

Приоритизация threads:
1. Thread, непосредственно связанный с текущей сценой.
2. Thread с последней активностью < 5 ходов назад.
3. Thread с наивысшим stakes.
4. Остальные — по дате последней активности.

**Формат:**

```
[АКТИВНЫЕ СЮЖЕТНЫЕ ЛИНИИ]

1. Тайна выживания короля (АКТИВНА)
   Ставки: Легитимность наследника. Если правда вскроется — гражданская война.
   Открытый вопрос: Почему монастырь на карантине?
   Последнее событие: Игрок решил отправиться к монастырю (ход 43).

2. Заговор Сафиры (АКТИВНА)
   Ставки: Контроль над Советом. Сафира готовит узурпацию.
   Открытый вопрос: Знает ли Лорд Кейран о заговоре?
```

### Слой 6: Mechanical State / Rules

**Приоритет:** НИЗКИЙ (но ВЫСОКИЙ в боевых/механических сценах).

**Содержимое:**

- Контекст текущего ruleset: только правила, релевантные текущей ситуации.
- Текущие механические эффекты: баффы, дебаффы, проклятия, яды, таймеры.
- Боевая ситуация (если активна): порядок инициативы, HP, позиции, доступные действия.
- Инвентарь игрока (или только ключевые предметы).
- Character sheet summary: только релевантные характеристики.

**Примерный размер:**

- Вне боя: 0–300 токенов (краткие правила + активные эффекты).
- В бою: 500–2000 токенов (полная боевая ситуация).

**Стратегия компрессии:**

- Вне боя: включить только активные эффекты и минимальные правила.
- В бою: приоритет слоя повышается до уровня Слоя 1 (обязательный). Боевая ситуация не может быть обрезана во время боя.
- Rules: включать только правила, релевантные текущему действию (обработка атаки — правила атаки; заклинание — правила магии).

**Динамическое повышение приоритета:**

В боевых сценах Слой 6 меняет свой приоритет:

| Состояние | Приоритет Слоя 6 |
|-----------|------------------|
| Exploration / social | НИЗКИЙ (после Слоя 5) |
| Pre-combat (инициатива объявлена) | СРЕДНИЙ (после Слоя 3) |
| Active combat | ВЫСОКИЙ (после Слоя 1, перед Слоем 2) |

**Формат:**

```
[МЕХАНИКА]

Система правил: D&D 5e (упрощённая)

Текущие эффекты:
- Игрок: Благословение (Bless), осталось 3 раунда.
- Лорд Кейран: Отравлен (−2 к атаке), до конца сцены.

Инвентарь (ключевое):
- Свиток Снятия Проклятия (1 шт.)
- Королевская печать (получена в ходе 31)
```

| Слой | Содержимое | Приоритет | Мин. размер | Макс. размер | Обрезаемый? |
|------|-----------|-----------|-------------|-------------|-------------|
| 0 | System prompt + instructions | ОБЯЗАТЕЛЬНЫЙ | 500 | 2000 | НЕТ |
| 1 | Текущая сцена | ОБЯЗАТЕЛЬНЫЙ | 100 | 1500 | Частично (сжимается) |
| 2 | NPC знания и цели | ВЫСОКИЙ | 0 | 3000 | ДА |
| 3 | Последние ходы | ВЫСОКИЙ | 300 (3 хода) | 10000 | ДА (старые ходы) |
| 4 | Retrieval results | СРЕДНИЙ | 0 | 3000 | ДА |
| 5 | Story threads | СРЕДНИЙ-НИЗКИЙ | 0 | 1400 | ДА |
| 6 | Rules / mechanical | НИЗКИЙ* | 0 | 2000 | ДА |

*Приоритет Слоя 6 повышается до ВЫСОКОГО в боевых сценах.

## 3. Multi-Pass Compilation

Context Compiler использует жадный алгоритм многопроходной компиляции. Каждый pass включает блоки одного слоя и обновляет оставшийся бюджет. Если бюджет исчерпан, оставшиеся pass'ы пропускаются.

### Общий алгоритм

```
Вход: max_context_budget, данные всех слоёв
Выход: compiled_context (список блоков), metadata (метрики)

remaining_budget = max_context_budget
compiled_blocks = []

for pass in [Pass1, Pass2, Pass3, Pass4, Pass5]:
    blocks, tokens_used = pass.execute(remaining_budget)
    compiled_blocks.extend(blocks)
    remaining_budget -= tokens_used
    if remaining_budget <= 0:
        break

return assemble(compiled_blocks), compute_metrics(compiled_blocks)
```

### Pass 1: Обязательные слои (Слой 0 + Слой 1)

**Цель:** Включить system prompt и описание текущей сцены.

**Алгоритм:**

1. Загрузить system prompt (Слой 0). Подсчитать токены.
2. Если system prompt > `remaining_budget` → **ОШИБКА** (context window слишком мал для работы). Вернуть ошибку, не пытаться генерировать.
3. Загрузить описание текущей сцены (Слой 1). Подсчитать токены.
4. Если `system_prompt + scene > remaining_budget` → сжать сцену до минимальной версии. Если всё ещё не помещается → **ПРЕДУПРЕЖДЕНИЕ**, включить только system prompt.
5. Обновить `remaining_budget`.

**Post-condition:** `remaining_budget >= 0`. System prompt всегда включён.

### Pass 2: NPC блоки (Слой 2)

**Цель:** Включить знания и цели присутствующих NPC.

**Алгоритм:**

1. Получить список NPC в текущей сцене, отсортированный по приоритету (см. приоритизацию NPC в описании Слоя 2).
2. Для каждого NPC, начиная с наиболее приоритетного:
   a. Сформировать полный NPC-блок. Подсчитать токены.
   b. Если полный блок помещается → включить.
   c. Если не помещается → попробовать сжатый блок.
   d. Если сжатый не помещается → попробовать минимальный блок.
   e. Если минимальный не помещается → пропустить этот и все последующие NPC.
3. Обновить `remaining_budget`.

**Правило:** Если NPC, с которым ведётся прямой диалог, не помещается даже в минимальном формате — это critical warning, потому что модель не будет знать, с кем разговаривает.

### Pass 3: Sliding Window (Слой 3)

**Цель:** Включить максимально возможное количество последних ходов.

**Алгоритм:**

1. Загрузить ходы из истории, начиная с последнего.
2. Для каждого хода (от нового к старому):
   a. Подсчитать токены хода (включая tool calls, если есть).
   b. Если ход помещается в `remaining_budget` → включить.
   c. Если не помещается → остановиться.
3. Если включено менее 3 ходов, попытаться освободить бюджет:
   a. Сжать NPC-блоки (Pass 2) до минимальных.
   b. Повторить попытку включения ходов.
4. Если даже после сжатия менее 3 ходов → **ПРЕДУПРЕЖДЕНИЕ**: модель не имеет достаточного контекста разговора.
5. Обновить `remaining_budget`.

**Правило минимума 3 ходов:** Три последних хода — это абсолютный минимум для связного продолжения диалога. Context Compiler должен гарантировать их включение даже ценой сжатия других слоёв.

### Pass 4: Retrieval Results (Слой 4)

**Цель:** Заполнить оставшееся пространство релевантными воспоминаниями из долговременной памяти.

**Алгоритм:**

1. Запросить retrieval pipeline: получить ранжированные memory chunks.
2. Отфильтровать chunks с `relevance_score < min_relevance_threshold`.
3. Дедуплицировать: если несколько chunks описывают одно событие, оставить один с максимальным score.
4. Для каждого chunk (от высокого relevance к низкому):
   a. Подсчитать токены.
   b. Если помещается → включить.
   c. Если не помещается → попробовать следующий (он может быть короче).
   d. Если оставшийся бюджет < минимального размера chunk'а → остановиться.
5. Обновить `remaining_budget`.

**Правило:** Retrieval results никогда не вытесняют обязательные слои или минимум ходов. Они заполняют только свободное пространство.

### Pass 5: Story Threads и Rules (Слои 5–6)

**Цель:** Заполнить оставшийся бюджет контекстом сюжетных линий и механик.

**Алгоритм:**

1. Определить текущий приоритет Слоя 6 (см. динамическое повышение приоритета).
2. Если боевая сцена: сначала включить Слой 6 (механика), затем Слой 5 (threads).
3. Если не боевая: сначала включить Слой 5 (threads), затем Слой 6 (механика).
4. Для каждого блока — стандартная процедура: подсчитать токены, включить если помещается, сжать если нет, пропустить если не помещается даже в сжатом виде.
5. Обновить `remaining_budget`.

### Диаграмма multi-pass compilation

```
┌─────────────────────────────────────────┐
│          max_context_budget             │
│  (model_window - response - safety)    │
└────────────────┬────────────────────────┘
                 │
    ┌────────────▼────────────┐
    │  Pass 1: System + Scene │
    │  (Слои 0 + 1)          │
    │  ОБЯЗАТЕЛЬНО            │
    └────────────┬────────────┘
                 │ remaining_budget
    ┌────────────▼────────────┐
    │  Pass 2: NPC Blocks     │
    │  (Слой 2)               │
    │  По приоритету NPC      │
    └────────────┬────────────┘
                 │ remaining_budget
    ┌────────────▼────────────┐
    │  Pass 3: Sliding Window │
    │  (Слой 3)               │
    │  От новых к старым      │
    │  Минимум 3 хода         │
    └────────────┬────────────┘
                 │ remaining_budget
    ┌────────────▼────────────┐
    │  Pass 4: Retrieval      │
    │  (Слой 4)               │
    │  По relevance score     │
    └────────────┬────────────┘
                 │ remaining_budget
    ┌────────────▼────────────┐
    │  Pass 5: Threads+Rules  │
    │  (Слои 5 + 6)           │
    │  Адаптивный приоритет   │
    └────────────┬────────────┘
                 │
    ┌────────────▼────────────┐
    │  Финальная сборка       │
    │  + подсчёт метрик       │
    └─────────────────────────┘
```

## 4. Формат инъекции знаний NPC

Как именно знания NPC представлены в контексте — один из ключевых архитектурных выборов. Он влияет на то, насколько точно модель будет играть NPC, и насколько надёжно она будет следовать ограничениям знаний.

### 4.1. Подход A: Structured XML-like Blocks

```xml
<npc name="Лиара" role="советница" emotional_state="настороженная">
  <knowledge>
    <fact confidence="0.9" source="письмо Сафиры">Король мёртв.</fact>
    <fact confidence="0.7" source="наблюдение">Лорд Кейран скрывает что-то о монастыре.</fact>
    <fact confidence="1.0" source="личный опыт">Сафира — верная подруга.</fact>
  </knowledge>
  <false_beliefs>
    <belief>Письмо Сафиры было подлинным.</belief>
  </false_beliefs>
  <does_not_know>
    <gap>Король жив и скрывается в монастыре.</gap>
    <gap>Письмо Сафиры подделано.</gap>
  </does_not_know>
  <goals>
    <goal priority="high">Найти законного наследника престола.</goal>
    <goal priority="medium">Понять, почему Кейран избегает темы монастыря.</goal>
  </goals>
  <relationships>
    <rel target="Игрок" trust="0.6" respect="0.8">Видит потенциального союзника, но ещё не полностью доверяет.</rel>
    <rel target="Лорд Кейран" trust="0.3" suspicion="0.7">Подозревает в сокрытии информации.</rel>
    <rel target="Сафира" trust="0.9" loyalty="0.8">Считает верной подругой и союзницей.</rel>
  </relationships>
  <secrets>
    <secret hidden_from="Лорд Кейран">Лиара тайно переписывается с Сафирой.</secret>
  </secrets>
</npc>
```

**Преимущества:**

- Чётко разделяет категории знаний — модели легче извлекать конкретные факты.
- Явный `<does_not_know>` блок: модель видит, чего NPC не знает, и реже «случайно» проговаривается.
- Удобен для machine parsing — если Scribe должен обновлять знания, structured формат проще парсить.
- Confidence и source видны модели — она может играть неуверенность.
- Лучше масштабируется: 5+ NPC в structured формате остаются читаемыми.

**Недостатки:**

- Занимает больше токенов из-за разметки (XML-теги — overhead ~15-20%).
- Может вызывать «роботизированный» стиль ответов у некоторых моделей.
- Требует от модели понимания XML-разметки (большинство современных LLM это умеют).

### 4.2. Подход B: Natural Language Summaries

```
Лиара — советница при дворе. Сейчас настороженна и напряжена.

Лиара считает короля мёртвым (уверенность высокая). Она узнала это из письма
Сафиры и полностью доверяет этому источнику. Она не знает, что письмо было
подделано, и не знает, что король жив и скрывается в монастыре.

Лиара подозревает, что Лорд Кейран что-то скрывает о монастыре. Она хочет
найти законного наследника престола и разобраться в поведении Кейрана.

Она видит в Игроке потенциального союзника, но ещё не полностью доверяет ему.
К Кейрану относится с подозрением. Сафиру считает верной подругой — и это
ошибка, о которой Лиара пока не знает.

Лиара тайно переписывается с Сафирой. Кейран об этом не знает.
```

**Преимущества:**

- Более естественный стиль → модель генерирует более нарративные ответы.
- Меньше overhead'а от разметки → больше полезных токенов.
- Легче читается в debug-логах для человека.
- Лучше подходит для моделей, оптимизированных под natural text (storytelling models).

**Недостатки:**

- Границы между категориями размыты — модель может «перемешать» знания и незнания.
- Сложнее для machine parsing при обновлении.
- При большом количестве NPC — сложнее различить, где заканчивается один NPC и начинается другой.
- `does_not_know` менее явный — модель может не обратить внимания на негативный контекст.

### 4.3. Рекомендация

| Задача | Рекомендуемый формат |
|--------|---------------------|
| Narrative generation (основной ход ДМа) | Natural language |
| Structured extraction (Memory Scribe) | XML-like |
| Tool use / function calling | XML-like |
| Маленькие модели (7B–13B) | XML-like (чёткие инструкции важнее) |
| Большие модели (70B+, GPT-4, Claude) | Natural language (лучше стиль) |
| Боевые сцены (механика важнее нарратива) | XML-like |
| Debug / troubleshooting | XML-like (проще анализировать) |

**Конфигурация:** Формат NPC-блоков должен быть настраиваемым:

```yaml
context_compiler:
  npc_format: "natural"  # "xml" | "natural" | "auto"
  # auto: xml для scribe/extraction, natural для narrative
```

В режиме `auto` Context Compiler выбирает формат в зависимости от текущей задачи, которую решает Turn Runner (narrative generation vs extraction).

## 5. Truncation Strategy

Когда общий объём данных всех слоёв превышает `max_context_budget`, Context Compiler должен обрезать контент. Порядок обрезки строго определён.

### 5.1. Порядок обрезки (от первого к последнему)

Обрезка происходит в следующем порядке — первым обрезается то, что имеет наименьшую ценность:

```
1. Rules / mechanical state (Слой 6)     ← режется первым
2. Story threads (Слой 5)
3. Retrieval results (Слой 4)
4. Старые ходы (Слой 3, от старых к новым)
5. NPC knowledge (Слой 2, от менее релевантных NPC)
6. Описание сцены (Слой 1, сжатие)      ← режется последним
7. System prompt (Слой 0)               ← НИКОГДА не режется
```

### 5.2. Правила обрезки по слоям

**Слой 6 (Rules):**
- Первым делом удаляются правила, не относящиеся к текущей ситуации.
- Затем удаляются неактивные эффекты.
- В боевой сцене: НЕ удаляется. Приоритет повышен. Обрезка начинается со Слоя 5.

**Слой 5 (Story threads):**
- Удаляются threads с `last_activity > 20 ходов назад`.
- Затем сжимаются оставшиеся threads до минимального формата.
- Затем удаляются по одному, начиная с наименее релевантного.

**Слой 4 (Retrieval results):**
- Удаляются chunks с наименьшим `relevance_score`.
- Порог удаления повышается (например, от 0.3 до 0.5, потом до 0.7).
- Последними удаляются chunks с `relevance_score > 0.8`.

**Слой 3 (Ходы):**
- Удаляются ходы от старых к новым.
- Удалённые ходы заменяются summary (если summary доступен).
- **Абсолютный минимум: 3 последних хода.** Эти ходы не могут быть удалены ни при каких обстоятельствах (кроме ситуации, когда даже system prompt + 3 хода не помещаются в бюджет — это критическая ошибка).

**Слой 2 (NPC knowledge):**
- Сначала сжимаются блоки NPC (полный → сжатый → минимальный).
- Затем удаляются целые NPC-блоки, начиная с наименее релевантных:
  1. NPC, не участвующий в диалоге и не упомянутый в последних ходах.
  2. NPC с наименьшим `importance_score`.
  3. NPC, не имеющий unresolved beats.
- **Последним удаляется** NPC, с которым ведётся прямой диалог. Его удаление — critical warning.

**Слой 1 (Сцена):**
- Полное описание → сжатое описание → минимальное (локация + присутствующие).
- Полное удаление сцены невозможно.

**Слой 0 (System prompt):**
- Никогда не обрезается и не модифицируется.
- Если system prompt не помещается в context window — ошибка конфигурации.

### 5.3. Cascading compression

Обрезка может быть каскадной. Если удаление одного слоя недостаточно, система продолжает обрезку следующего уровня:

```python
def truncate_to_budget(layers, budget):
    """
    Каскадная обрезка слоёв до тех пор, пока суммарный размер не уместится в бюджет.
    """
    total = sum(layer.token_count for layer in layers)
    
    if total <= budget:
        return layers  # Обрезка не нужна
    
    # Порядок обрезки
    truncation_order = [
        (Layer6, "remove"),    # Удалить rules
        (Layer5, "compress"),  # Сжать threads
        (Layer5, "remove"),    # Удалить threads
        (Layer4, "trim_low"),  # Удалить низко-релевантные retrieval
        (Layer4, "remove"),    # Удалить все retrieval
        (Layer3, "trim_old"),  # Удалить старые ходы
        (Layer2, "compress"),  # Сжать NPC блоки
        (Layer2, "trim_low"),  # Удалить неприоритетных NPC
        (Layer1, "compress"),  # Сжать сцену
    ]
    
    for layer, action in truncation_order:
        apply_truncation(layer, action)
        total = sum(l.token_count for l in layers)
        if total <= budget:
            return layers
    
    # Если после всех обрезок не помещается — critical error
    raise ContextBudgetExceededError(
        f"Cannot fit mandatory content ({total} tokens) "
        f"into budget ({budget} tokens)"
    )
```

## 6. Отладка и Memory Inspector

Context Compiler генерирует огромное количество неявных решений на каждый ход: что включить, что обрезать, какой NPC получит полный блок, а какой — минимальный. Без инструментов отладки и визуализации невозможно понять, почему модель сгенерировала определённый ответ.

### 6.1. Логирование финального контекста

Каждый вызов Context Compiler должен сохранять полный snapshot скомпилированного контекста. Snapshot является частью Turn record (см. product-foundation.md, секция 10, Turn: «использованный контекст»).

```yaml
      chunks_considered: 15
      chunks_rejected: 8
      min_relevance_included: 0.45
      max_relevance_rejected: 0.42
    - layer: 5
      name: "Story threads"
      tokens: 600
      threads_included: 3
      threads_total: 5
      compression_level: "compressed"
    - layer: 6
      name: "Rules"
      tokens: 200
      truncated: true
      compression_level: "minimal"
  compilation_time_ms: 47
  warnings:
    - "Layer 6 truncated to minimal due to budget constraints"
```

### 6.2. Визуализация в Memory Inspector

Memory Inspector (UI компонент, см. product-foundation.md, секция 9, UI → Memory Inspector) должен предоставлять для Context Compiler:

**Основной вид: Token Budget Breakdown**

Визуальная диаграмма (stacked bar или treemap), показывающая:

- Общий бюджет (ширина всей полосы).
- Долю каждого слоя (цветовые сегменты).
- Неиспользованный бюджет (серый сегмент).
- Зарезервированные для ответа токены (отдельная отметка).

```
[=== System (1.2k) ===|== Scene (0.9k) ==|=== NPC (2.4k) ===|======== Turns (18.2k) ========|= Retr (2.9k) =|St(0.6k)|R|_rem_]
                                                                                                                             ↑
                                                                                                              remaining: 2.6k
```

**Детальный вид: Layer Drill-Down**

При клике на сегмент слоя — детализация:

- Слой 2 (NPC): список NPC с уровнем компрессии, размером блока, кнопкой «показать содержимое».
- Слой 3 (Ходы): список включённых ходов, визуальная индикация обрезанного диапазона.
- Слой 4 (Retrieval): список chunks с relevance score, source, preview текста. Отброшенные chunks показываются отдельно (серые, с причиной отклонения).

**Вид сравнения: Diff View**

Показать разницу между контекстами двух последовательных ходов:

- Что было добавлено.
- Что было удалено.
- Какие NPC-блоки изменились.
- Как изменился sliding window.

### 6.3. Replay хода с другим контекстом

Memory Inspector должен поддерживать функцию «What-If Replay»:

1. Пользователь выбирает ход из истории.
2. Система показывает контекст, который был скомпилирован для этого хода (из snapshot).
3. Пользователь может модифицировать контекст:
   - Добавить/удалить NPC-блоки.
   - Изменить retrieval results.
   - Добавить/убрать story threads.
   - Изменить бюджет.
4. Пользователь нажимает «Replay» — модифицированный контекст отправляется модели.
5. Результат показывается рядом с оригинальным ответом для сравнения.

**Use cases:**

- Дебаг: «Если бы NPC знал X, как бы изменился ответ?»
- Тюнинг: «Если уменьшить бюджет на 50%, как сильно пострадает качество?»
- Обучение: понимание, как контекст влияет на поведение модели.

## 7. Метрики

Context Compiler должен собирать метрики на каждый вызов. Метрики используются для:

- Мониторинга здоровья системы.
- Оптимизации параметров (бюджет, пороги, стратегии).
- Диагностики проблем качества ответов.
- Отображения в Memory Inspector.

### 7.1. Основные метрики

| Метрика | Формула | Целевое значение | Описание |
|---------|---------|------------------|----------|
| `utilization_rate` | `used_tokens / max_budget` | 85–95% | Насколько полно используется бюджет. <80% — модель не получает достаточно контекста. >95% — слишком близко к лимиту, высокий риск обрезки. |
| `retrieval_relevance_avg` | `mean(chunk.relevance_score)` для включённых chunks | >0.5 | Средняя релевантность retrieval results. Низкое значение означает, что долговременная память не находит ничего полезного. |
| `retrieval_relevance_min` | `min(chunk.relevance_score)` для включённых chunks | >0.3 | Минимальная релевантность — индикатор «мусора» в контексте. |
| `npc_coverage` | `included_npcs / present_npcs` | 1.0 | Доля NPC из текущей сцены, включённых в контекст. <1.0 означает, что модель не знает о некоторых NPC в сцене. |
| `npc_full_coverage` | `full_npcs / included_npcs` | >0.5 | Доля NPC, получивших полный (несжатый) блок знаний. |
| `turn_window_depth` | Количество включённых ходов | ≥3, целевое: 10–30 | Глубина скользящего окна. Менее 3 — critical. |
| `turn_window_ratio` | `included_turns / total_turns` | Зависит от истории | Какую долю всей истории видит модель через sliding window. |
| `layers_included` | Битовая маска включённых слоёв | Все | Какие слои были включены. Пропуск слоя — сигнал о нехватке бюджета. |
| `layers_truncated` | Список слоёв, подвергшихся обрезке | Пусто | Какие слои были обрезаны. |
| `compilation_time_ms` | Время работы Context Compiler в мс | <200ms | Время от начала compilation до готового контекста. Включает retrieval, token counting, assembly. |
| `retrieval_time_ms` | Время retrieval pipeline в мс | <100ms | Подсоставная часть compilation_time. |
| `token_counting_time_ms` | Время подсчёта токенов в мс | <50ms | Подсоставная часть compilation_time. |

### 7.2. Агрегированные метрики (за сессию / кампанию)

| Метрика | Описание |
|---------|----------|
| `avg_utilization_rate` | Средняя утилизация за сессию. Трендовый индикатор. |
| `truncation_frequency` | Как часто происходит обрезка (% ходов). Высокая частота → нужно увеличить бюджет или оптимизировать контент. |
| `npc_exclusion_frequency` | Как часто NPC исключаются. Высокая частота → проблема с количеством NPC в сценах. |
| `min_turn_window_depth` | Минимальная глубина window за сессию. Если = 3 — бюджет был на грани. |
| `retrieval_hit_rate` | В скольких ходах retrieval вернул хотя бы один chunk с score > 0.5. |

### 7.3. Алерты

Система должна генерировать предупреждения при следующих условиях:

| Условие | Уровень | Описание |
|---------|---------|----------|
| `utilization_rate > 0.98` | WARNING | Бюджет почти исчерпан, минимальный запас. |
| `utilization_rate < 0.5` | INFO | Бюджет недоиспользован — либо мало данных, либо слишком большой context window. |
| `turn_window_depth < 3` | CRITICAL | Модель не видит достаточно истории для связного ответа. |
| `npc_coverage < 1.0` | WARNING | Некоторые NPC в сцене не представлены в контексте. |
| `compilation_time_ms > 500` | WARNING | Компиляция слишком долгая, может влиять на UX. |
| `retrieval_relevance_avg < 0.3` | INFO | Retrieval не находит ничего полезного — возможно, нужна переиндексация. |
| System prompt > 50% budget | WARNING | System prompt занимает слишком много места — слишком мало остаётся для динамического контента. |

## 8. Edge Cases

### 8.1. Очень маленькое context window (4k–8k)

**Модели:** Мелкие локальные модели, старые версии, некоторые fine-tuned модели.

**Проблема:** После system prompt (1–2k) и reserved_for_response (1k) остаётся 1–5k токенов на весь динамический контекст.

**Стратегия:**

1. System prompt должен быть максимально компактным. Предусмотреть «short» версию system prompt (<500 токенов).
2. NPC-блоки — только минимальный формат (имя + одно предложение).
3. Sliding window — строго 3 хода, не более.
4. Retrieval — 0–1 chunk, только при relevance > 0.7.
5. Story threads — полностью исключаются.
6. Rules — только если активный бой, и даже тогда — минимальный формат.
7. Сцена — минимальный формат (локация + имена).

**Конфигурация:**

```yaml
# Preset для маленьких моделей
small_context_preset:
  reserved_for_response: 512
  safety_margin_percent: 10  # Увеличен — погрешность выше
  system_prompt_variant: "short"
  max_turn_window_depth: 5
  max_retrieval_chunks: 1
  min_relevance_threshold: 0.7
  include_story_threads: false
  npc_max_compression: "minimal"
```

**Ожидание:** Качество ответов будет значительно ниже. Система должна предупреждать пользователя:

> «Context window вашей модели (4096 токенов) сильно ограничивает возможности системы. Рекомендуется модель с context window ≥ 16k токенов для полноценной игры.»

### 8.2. Очень большое context window (128k+)

**Модели:** GPT-4 Turbo (128k), Claude 3 (200k), Gemini Pro (1M+), некоторые fine-tuned Llama 3.1 (128k).

**Проблема:** Избыток места может привести к:

- Включению слишком большого количества нерелевантного контента.
- Высокой стоимости API-вызовов (платится за input tokens).
- «Потерянной середине» (lost-in-the-middle): модели хуже обрабатывают информацию в середине длинного контекста.
- Увеличенной latency.

**Стратегия:**

1. **Не заполнять бюджет ради заполнения.** Utilization rate 60–80% допустим и даже предпочтителен для 128k+ моделей.
2. **Увеличить sliding window** — до 50–100 ходов, чтобы модель видела больше истории.
3. **Увеличить retrieval** — больше chunks, можно понизить threshold.
4. **Включить полные NPC-блоки** для всех NPC, включая не присутствующих, но упомянутых.
5. **Включить полные story threads** — все, не только активные.
6. **НО:** установить `max_context_budget` ниже физического лимита для экономии:

```yaml
# Preset для больших моделей
large_context_preset:
  manual_budget_override: 32768  # Ограничить 32k для экономии
  # ИЛИ: использовать адаптивный бюджет, но с мягким лимитом
  soft_budget_limit: 65536  # Рекомендуемый максимум
  reserved_for_response: 4096
  max_turn_window_depth: 100
  max_retrieval_chunks: 15
  min_relevance_threshold: 0.2
  include_story_threads: true
  npc_format: "natural"  # Больше места — можно позволить естественный формат
```

7. **Структурирование:** При длинном контексте критично использовать чёткие разделители между секциями, чтобы модель могла ориентироваться:

```
=== SYSTEM INSTRUCTIONS ===
...
=== CURRENT SCENE ===
...
=== NPC: ЛИАРА ===
...
=== NPC: ЛОРД КЕЙРАН ===
...
=== DIALOGUE HISTORY ===
...
=== RELEVANT MEMORIES ===
...
=== STORY THREADS ===
...
```

### 8.3. Первый ход кампании

**Проблема:** При первом ходе отсутствуют:

- История ходов (Слой 3 пуст).
- Retrieval results (Слой 4 пуст — нечего искать).
- Story threads (Слой 5 пуст — ещё нет сюжетных линий).
- Accumulated NPC knowledge (Слой 2 минимален — только начальные знания).

**Стратегия:**

1. Слой 0 и Слой 1 заполняются нормально — system prompt и стартовая сцена.
2. Слой 2: начальные NPC-блоки из campaign setup (пользователь создал NPC до начала игры).
3. Слой 3: пуст. Это нормально.
4. Слой 4: пуст. Это нормально.
5. Слой 5: если пользователь задал начальные story hooks — включить.
6. Слой 6: базовые правила, если ruleset подключён.
7. **Дополнительно:** Можно включить расширенный world lore / setting description, который обычно не помещается. Первый ход — единственный момент, когда бюджет позволяет включить детальное описание мира.

**Конфигурация:**

```yaml
first_turn_extras:
  include_world_lore: true
  include_setting_description: true
  include_campaign_backstory: true
  max_world_lore_tokens: 3000
```

**Ожидание:** `utilization_rate` на первом ходу будет низким (30–60%). Это нормально. Система не должна генерировать warning об недоиспользовании бюджета на первых 5 ходах.

### 8.4. Массовая сцена (10+ NPC)

**Проблема:** 10+ NPC × 400 токенов = 4000+ токенов только на NPC-блоки. При ограниченном бюджете это вытесняет ходы и retrieval.

**Стратегия:**

1. **Tier-система NPC:**

| Tier | Описание | Формат | Токенов |
|------|----------|--------|---------|
| A | Прямой участник диалога | Полный | 400–600 |
| B | Упомянутый / с unresolved beat | Сжатый | 150–200 |
| C | Просто присутствует | Минимальный | 30–50 |
| D | Фоновый / толпа | Только имя | 5–10 |

2. **Группировка:** Однородные NPC (стражники, горожане, культисты) можно объединить:

```
Присутствуют также: 6 стражников (лояльны Кейрану, не знают о заговоре, 
выполняют приказы без вопросов) и 4 горожанина-просителя (испуганы, 
ищут защиты от набегов).
```

Это ~50 токенов вместо 6×50 = 300 токенов.

3. **Динамическое переключение тиров:** Если игрок обращается к ранее фоновому NPC, его tier повышается на следующем ходу.

4. **Лимит полных блоков:** `max_full_npc_blocks` (по умолчанию 5). Даже если бюджет позволяет — больше 5 полных NPC-блоков создают confusion для модели.

5. **Scene summary для толпы:**

```
[СЦЕНА: ТРОННЫЙ ЗАЛ — БОЛЬШОЙ СОВЕТ]
В зале около 20 человек. Ключевые участники описаны отдельно.
Остальные: военачальники, советники, слуги. Большинство нервничают 
перед битвой. Никто из присутствующих не знает о заговоре Сафиры.
```

### 8.5. Переход между сценами

**Проблема:** При смене сцены нужно:

- Загрузить описание новой локации.
- Загрузить NPC новой сцены (могут быть совершенно другие).
- Сохранить контекст перехода (почему игрок здесь, что происходило до).
- Не потерять незавершённые действия из предыдущей сцены.

**Стратегия:**

1. Включить summary предыдущей сцены (50–150 токенов) в описание текущей сцены (Слой 1).
2. NPC-блоки полностью перезагружаются для новой сцены.
3. Retrieval может быть нацелен на события, связанные с новой локацией и новыми NPC.
4. Sliding window сохраняется — последние ходы содержат момент перехода.

### 8.6. Развилка кампании (Branch)

**Проблема:** При ветвлении (branch) контекст должен соответствовать состоянию выбранной ветки.

**Стратегия:**

1. Все фильтры retrieval включают `branch_id`.
2. NPC-блоки загружаются из состояния ветки.
3. Sliding window берётся из истории ходов ветки.
4. Story threads фильтруются по ветке.
5. При переключении между ветками — полная перекомпиляция контекста.

## 9. Порядок блоков в финальном промпте

Порядок расположения блоков в скомпилированном промпте влияет на внимание модели. Современные LLM лучше обрабатывают начало и конец контекста (primacy/recency effect), хуже — середину.

### Рекомендуемый порядок

```
1. [SYSTEM]          — System prompt + instructions      (начало — высокое внимание)
2. [ТЕКУЩАЯ СЦЕНА]   — Сцена, локация, атмосфера        (близко к началу)
3. [NPC]             — Блоки NPC                         (близко к началу)
4. [СЮЖЕТНЫЕ ЛИНИИ]  — Story threads                     (середина — ниже внимание)
5. [ВОСПОМИНАНИЯ]    — Retrieval results                 (середина)
6. [МЕХАНИКА]        — Rules / combat state              (середина)
7. [ДИАЛОГ]          — Sliding window (ходы)             (конец — высокое внимание)
```

**Обоснование:**

- System prompt в начале — стандартная практика, модели оптимизированы для этого.
- Сцена и NPC близко к началу — модель хорошо запоминает «кто мы» и «где мы».
- Story threads и retrieval в середине — они информационно важны, но менее критичны для непосредственного ответа.
- Диалог (ходы) в конце — recency bias работает в нашу пользу. Модель лучше всего помнит последние сообщения, что обеспечивает связность ответа с последним ходом игрока.

### Альтернативный порядок для боевых сцен

```
1. [SYSTEM]
2. [МЕХАНИКА]        — Боевые правила и состояние (критично в бою)
3. [ТЕКУЩАЯ СЦЕНА]   — Позиции, инициатива
4. [NPC]             — Только боевые характеристики
5. [ДИАЛОГ]          — Последние 3-5 ходов боя
```

В бою нарратив уступает место точности механик, поэтому правила и состояние боя получают primacy position.

## 10. Интерфейс Context Compiler

### 10.1. Входной контракт

```python
@dataclass
class CompilationRequest:
    """Запрос на компиляцию контекста для одного хода."""
    
    campaign_id: str
    branch_id: str
    scene: Scene
    turn_history: list[Turn]            # Полная история ходов (compiler сам обрежет)
    present_npcs: list[Character]       # NPC в текущей сцене
    npc_knowledge: dict[str, NPCKnowledgeBlock]  # Знания, цели, отношения каждого NPC
    active_threads: list[StoryThread]
    mechanical_state: MechanicalState | None
    retrieval_results: list[MemoryChunk]  # Уже ранжированные retrieval results
    
    # Конфигурация
    budget_config: BudgetConfig
    npc_format: Literal["xml", "natural", "auto"]
    task_type: Literal["narrative", "extraction", "combat"]
    debug_mode: bool = False
```

### 10.2. Выходной контракт

```python
@dataclass
class CompilationResult:
    """Результат компиляции контекста."""
    
    # Готовый контекст для отправки модели
    messages: list[Message]  # В формате chat messages (system, user, assistant)
    
    # Метаданные
    metadata: CompilationMetadata
    
    # Debug информация (только если debug_mode=True)
    debug: CompilationDebug | None
    
    # Предупреждения
    warnings: list[CompilationWarning]

@dataclass
class CompilationMetadata:
    total_tokens: int
    max_budget: int
    utilization_rate: float
    layers_included: list[int]       # [0, 1, 2, 3, 4, 5, 6]
    layers_truncated: list[int]      # [5, 6]
    turn_window_depth: int
    npcs_included: int
    npcs_total: int
    retrieval_chunks_included: int
    retrieval_chunks_total: int
    compilation_time_ms: int
```

## 11. Связь с другими компонентами

Context Compiler не работает в изоляции. Он является центральным узлом, получающим данные от множества компонентов и передающим результат Turn Runner'у.

```
                    ┌──────────────┐
                    │  Turn Runner │
                    │  (вызывает)  │
                    └──────┬───────┘
                           │
                           ▼
┌──────────────────────────────────────────────────┐
│                CONTEXT COMPILER                   │
│                                                   │
│  Получает от:                Отдаёт:             │
│  ├─ Scene Director           ├─ Compiled Context │
│  │  (текущая сцена)         ├─ Metadata          │
│  ├─ Campaign Service         ├─ Debug Info        │
│  │  (config, NPC, threads)  └─ Warnings          │
│  ├─ Memory Scribe                                │
│  │  (NPC knowledge blocks)                       │
│  ├─ Retrieval Pipeline                           │
│  │  (ranked memory chunks)                       │
│  ├─ Rules Dispatcher                             │
│  │  (mechanical state)                           │
│  └─ Turn/Event Log                               │
│     (turn history)                               │
└──────────────────────────────────────────────────┘
                           │
                           ▼
                    ┌──────────────┐
                    │   LLM Call   │
                    │ (via Provider│
                    │   Gateway)   │
                    └──────────────┘
```

**Важно:** Context Compiler НЕ вызывает retrieval pipeline напрямую. Retrieval вызывается Turn Runner'ом до компиляции и передаёт результаты в `CompilationRequest.retrieval_results`. Это разделение ответственности позволяет:

- Тестировать Context Compiler без зависимости от vector DB.
- Подставлять фиктивные retrieval results для replay и debug.
- Кешировать retrieval отдельно от компиляции.

## 12. Открытые вопросы

1. **Адаптивный reserved_for_response:** Должен ли Turn Runner динамически менять размер зарезервированного для ответа пространства в зависимости от типа хода (длинное описание vs короткая реплика)?

2. **Cross-scene NPC knowledge:** Если NPC не присутствует в сцене, но упомянут игроком — нужно ли включать его knowledge block? Если да, в каком приоритете?

3. **Summarization chain:** Должен ли Context Compiler сам суммаризировать старые ходы, или это задача Memory Scribe, а compiler только использует готовые summary?

4. **Multi-model context:** При разных моделях для narrative и extraction — нужны ли разные компиляции? Скорее всего да, что удваивает вызовы compiler'а.

5. **Prompt caching:** Некоторые provider'ы (Anthropic) поддерживают prompt caching. Context Compiler должен учитывать это: ставить неизменные блоки (system prompt, NPC blocks) в начало, чтобы они попадали в кеш.

6. **Streaming compilation:** Можно ли начать генерацию до завершения retrieval, отправив обязательные слои сразу? Технически возможно с function calling, но сильно усложняет архитектуру.

7. **Quality feedback loop:** Как измерить, что контекст был «хорошим»? Нужна ли метрика качества ответа, привязанная к контексту? Потенциальный подход: user thumbs up/down → корреляция с параметрами компиляции.
