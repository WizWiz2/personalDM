## 1. Обзор

Этот документ детализирует две критические подсистемы, обозначенные в product-foundation.md, но не раскрытые в достаточной мере:

1. **Memory Compaction** — стратегия управления объёмом памяти: как сырой архив трансформируется в оперативную память и memory chunks, не теряя важные данные.
2. **Memory Scribe FSM** — конечный автомат жизненного цикла записей, предложенных Memory Scribe: от предложения через одобрение до архивации или отклонения.

Оба механизма критичны для выполнения обещания product-foundation.md: система должна работать с кампаниями в сотни и тысячи ходов, не теряя канон и не перегружая context window модели.

## 2. Memory Compaction

### 2.1. Общая схема уровней

```
┌──────────────────────────────────────────────────┐
│  RAW ARCHIVE (append-only, immutable)            │
│  Каждый Turn, каждый ответ, каждый tool call     │
│  Неограниченный рост. Никогда не модифицируется. │
└────────────────────┬─────────────────────────────┘
                     │ периодическая суммаризация
                     ▼
┌──────────────────────────────────────────────────┐
│  MEMORY CHUNKS (derived, rebuildable)            │
│  Суммаризированные блоки по сценам/периодам      │
│  Retention policy: важные живут дольше           │
└────────────────────┬─────────────────────────────┘
                     │ context compilation
                     ▼
┌──────────────────────────────────────────────────┐
│  OPERATIVE MEMORY (sliding window)               │
│  То, что реально уходит в prompt LLM            │
│  Адаптивный размер под context window модели     │
└──────────────────────────────────────────────────┘
```

### 2.2. Сырой архив (Raw Archive)

**Принцип:** append-only, никогда не модифицируется.

Сырой архив — это фундамент, из которого всё остальное может быть пересобрано. В терминах event sourcing, это event log.

Содержимое каждой записи Turn (см. §10 product-foundation.md):
- Полный текст хода пользователя
- Полный текст ответа DM
- Tool calls и их результаты
- State delta (какие изменения были применены)
- Использованный контекст (snapshot или ссылка)
- Версия модели и промпта
- Метаданные: timestamp, scene_id, branch_id, session_id

**Рост:**

| Параметр | Оценка |
|----------|--------|
| Средний размер Turn (user + DM + meta) | ~2-5 KB |
| 500 ходов | ~1-2.5 MB |
| 1000 ходов | ~2-5 MB |
| 5000 ходов | ~10-25 MB |

Это вполне терпимо для SQLite. Проблема не в хранении, а в том, что весь архив не помещается в context window модели.

**Ключевое правило:** никакие операции (undo, rollback, branching, memory correction) не модифицируют записи сырого архива. Они могут добавлять новые записи (например, "correction applied at turn N") или помечать записи метаданными, но не изменять текст.

### 2.3. Оперативная память (Sliding Window)

**Назначение:** формирование блока «последние ходы» для prompt LLM.

#### 2.3.1. Размер окна: token count vs turn count

Два подхода, каждый с недостатками:

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| **Turn count** (фиксированное число ходов) | Простота реализации; предсказуемость; одинаковое поведение на любой модели | Ходы разной длины → непредсказуемое заполнение context window; длинный ход может взорвать контекст |
| **Token count** (бюджет в токенах) | Точный контроль; максимальное использование context window | Нужен tokenizer или аппроксимация; разные модели считают токены по-разному; зависимость от модели |

**Рекомендуемый подход: гибридный**

1. Основной бюджет задаётся в **токенах** (или символах с коэффициентом конвертации ~4 символа ≈ 1 токен для английского, ~2-3 символа ≈ 1 токен для русского).
2. Задаётся **минимальное количество ходов** (например, 3), которые включаются всегда, даже если превышают бюджет.
3. Задаётся **максимальное количество ходов** (например, 50), чтобы очень короткие ходы не тащили бесконечный хвост.

Формула:
```
effective_window = max(
  min_turns,
  min(
    max_turns,
    turns_fitting_in(token_budget - reserved_for_system_prompt - reserved_for_context)
  )
)
```

#### 2.3.2. Бюджетирование context window

Context window модели не принадлежит целиком sliding window. Он делится между несколькими потребителями:

```
┌──────────────────────────────────────────────────┐
│              TOTAL CONTEXT WINDOW                │
│                                                  │
│  ┌───────────────────────────────────────────┐   │
│  │ System Prompt + Campaign Instructions     │   │
│  │ (~500-2000 tokens)                        │   │
│  ├───────────────────────────────────────────┤   │
│  │ Current Scene Description                 │   │
│  │ (~200-500 tokens)                         │   │
│  ├───────────────────────────────────────────┤   │
│  │ Character Sheets (present characters)     │   │
│  │ (~100-400 tokens per character)           │   │
│  ├───────────────────────────────────────────┤   │
│  │ Relevant Memory Chunks (retrieved)        │   │
│  │ (~500-2000 tokens)                        │   │
│  ├───────────────────────────────────────────┤   │
│  │ Active Facts & Relationships              │   │
│  │ (~200-1000 tokens)                        │   │
│  ├───────────────────────────────────────────┤   │
│  │ ★ SLIDING WINDOW (recent turns) ★        │   │
│  │ (всё оставшееся пространство)            │   │
│  ├───────────────────────────────────────────┤   │
│  │ Reserved for Generation                   │   │
│  │ (~500-2000 tokens)                        │   │
│  └───────────────────────────────────────────┘   │
└──────────────────────────────────────────────────┘
```

Context Compiler (§11.1) должен:
1. Вычислить бюджеты каждого блока
2. Заполнить фиксированные блоки (system prompt, scene, characters)
3. Заполнить retrieved memory chunks
4. Отдать остаток sliding window

**Приоритеты при нехватке места:**
1. System prompt — никогда не урезается
2. Текущая сцена — никогда не урезается
3. Минимальные последние ходы (min_turns) — никогда не урезаются
4. Присутствующие персонажи — можно урезать до essential traits
5. Retrieved memories — можно уменьшить количество
6. Остальные ходы sliding window — заполняют оставшееся

#### 2.3.3. Смена модели (изменение context window)

Когда пользователь меняет модель (например, с 8K на 128K или наоборот), система должна:

1. **Пересчитать бюджеты** всех блоков Context Compiler
2. **Не мигрировать данные** — sliding window вычисляется динамически при каждом ходе
3. **Уведомить пользователя** при значительном сокращении контекста:
   - «Новая модель имеет context window 8K (было 32K). Количество доступных ходов в контексте сократится с ~30 до ~8.»
4. **Записать в метаданные** факт смены модели (для отладки и аналитики)

Это работает автоматически, потому что sliding window — это runtime view, а не персистентное состояние. Ничего не нужно перезаписывать.

#### 2.3.4. Суммаризация выпавших ходов

Когда ходы выпадают из sliding window, они не просто исчезают. Система должна обеспечить плавный переход:

```
[Summary of turns 1-45]      ← суммаризация старых ходов
[Turn 46 full text]           ← начало sliding window
[Turn 47 full text]
...
[Turn 60 full text]           ← текущий ход
```

Суммаризация старых ходов — это задача Memory Chunks (см. 2.4).

### 2.4. Memory Chunks

**Назначение:** периодически суммаризировать блоки ходов в компактные поисковые единицы.

#### 2.4.1. Триггеры суммаризации

Суммаризация может срабатывать по трём триггерам (OR-логика):

| Триггер | Описание | Пример |
|---------|----------|--------|
| **По количеству ходов** | Каждые N ходов, не покрытых chunk'ом | Каждые 10-20 ходов |
| **По смене сцены** | При переходе из одной сцены в другую | Завершение сцены "Тронный зал" |
| **По времени (wall clock)** | При возобновлении сессии после длительного перерыва | Пользователь вернулся через 3 дня |

**Рекомендуемый primary trigger: смена сцены.** 

Сцена — естественная единица нарратива. Chunk, соответствующий одной сцене, имеет чёткие границы и семантическую целостность. Fallback: если сцена длится слишком долго (>30 ходов без смены), создавать промежуточный chunk.

#### 2.4.2. Процесс создания chunk

```
1. Определить span: какие ходы покрываются
2. Собрать полный текст ходов из raw archive
3. Собрать structured deltas (facts, relationships, events) за этот span
4. Отправить LLM-суммаризатору:
   - "Суммаризируй следующий блок игры. Сохрани ключевые события,
     решения, раскрытую информацию, изменения отношений."
5. Получить текстовый summary
6. Вычислить embedding для vector search
7. Извлечь keywords и entity IDs
8. Записать Memory Chunk в БД
```

**Важно:** для суммаризации можно использовать другую (более дешёвую/быструю) модель, чем для нарратива. Это то разделение narrator/scribe, о котором говорится в §22 product-foundation.md.

#### 2.4.3. Иерархическая суммаризация

При длинных кампаниях одноуровневой суммаризации недостаточно. Нужна иерархия:

```
Level 0: Raw turns                    (полный текст)
Level 1: Scene chunks                 (summary 10-30 ходов → ~200-500 tokens)
Level 2: Arc chunks                   (summary 5-10 сцен → ~200-500 tokens)
Level 3: Campaign epoch chunks        (summary 3-5 арок → ~200-500 tokens)
```

**Когда создаются chunks уровня 2+:**

| Уровень | Триггер |
|---------|---------|
| L1 (Scene) | Смена сцены или каждые N ходов |
| L2 (Arc) | Каждые 5-10 L1 chunks или ручная разметка «арки» |
| L3 (Epoch) | Каждые 3-5 L2 chunks или значимое событие кампании |

В MVP достаточно L0 + L1. L2 добавить, когда кампании начнут превышать 200-300 ходов.

#### 2.4.4. Merge chunks одной сцены

Если сцена длинная и породила несколько промежуточных chunks, при завершении сцены:

1. Собрать все L1 chunks этой сцены
2. Отправить LLM: «Объедини эти резюме в единое scene summary»
3. Пометить промежуточные chunks как `superseded_by: scene_summary_chunk_id`
4. Новый chunk наследует максимальный `importance` из исходных
5. Superseded chunks исключаются из retrieval, но не удаляются (для аудита)

#### 2.4.5. Retention Policy

Не все chunks равноценны. Важные должны жить дольше и чаще попадать в контекст.

**Факторы importance:**

| Фактор | Вес | Пример |
|--------|-----|--------|
| Содержит death/major event | Высокий | Смерть ключевого NPC |
| Содержит secret reveal | Высокий | Раскрытие предательства |
| Содержит relationship change | Средний | Союз стал враждой |
| Содержит new quest/goal | Средний | Получение задания |
| Обычный exploration/dialogue | Низкий | Осмотр таверны |
| Пользователь вручную отметил как важное | Максимальный | — |

**Retention tiers:**

| Importance | Retention | Поведение при сжатии |
|------------|-----------|---------------------|
| Critical | Permanent | Никогда не мержится в higher-level chunk без сохранения |
| High | Long | Мержится в L2 только после 200+ ходов |
| Normal | Standard | Мержится в L2 по обычному расписанию |
| Low | Short | Мержится агрессивно, может быть fully superseded |

**Важное ограничение:** retention policy влияет только на то, как быстро chunk мержится в более высокий уровень. Сам chunk никогда не удаляется из БД — он помечается как superseded, и его данные всегда можно восстановить из raw archive.

### 2.5. Полная переиндексация (Rebuild)

**Назначение:** пересборка всех derived данных (chunks, embeddings, indexes) из сырого архива.

#### 2.5.1. Когда нужна переиндексация

| Сценарий | Обоснование |
|----------|-------------|
| Смена embedding модели | Старые embeddings несовместимы с новой моделью |
| Обнаружение corrupted данных | Chunk содержит неправильную суммаризацию |
| Major version migration | Изменилась схема Memory Chunk |
| Пользователь массово исправил факты | Chunks содержат устаревшую информацию |
| Переход SQLite → PostgreSQL | Миграция storage layer |
| Debug / troubleshooting | Воспроизведение проблемы |

#### 2.5.2. Процесс rebuild

```
1. Остановить фоновые процессы (indexing, summarization)
2. Пометить все derived данные как stale
3. Итерировать по raw archive в хронологическом порядке:
   a. Для каждого Turn — пересоздать structured deltas
   b. Для каждого scene boundary — пересоздать L1 chunk
   c. Для каждого arc boundary — пересоздать L2 chunk
   d. Для каждого chunk — пересчитать embedding
4. Пересобрать FTS5 индекс
5. Пересобрать vector индекс
6. Валидировать целостность
7. Удалить stale данные
8. Возобновить фоновые процессы
```

#### 2.5.3. Оценка времени

| Кампания | Шаг | Оценка |
|----------|-----|--------|
| 500 ходов | Чтение raw archive | <1 сек |
| 500 ходов | Суммаризация (~25 L1 chunks) | ~2-5 мин (LLM) |
| 500 ходов | Embedding (25 chunks + entities) | ~10-30 сек (локальная модель) |
| 500 ходов | FTS + vector reindex | ~5-10 сек |
| **500 ходов** | **Итого** | **~3-6 мин** |
| 1000 ходов | Итого | ~6-12 мин |
| 5000 ходов | Итого | ~30-60 мин |

Bottleneck — LLM-суммаризация. Можно ускорить:
- Параллельной суммаризацией (chunks независимы)
- Использованием быстрой/мелкой модели для суммаризации
- Кэшированием: если raw data не изменились, chunk можно не пересоздавать

#### 2.5.4. Partial rebuild

Полный rebuild не всегда нужен. Варианты частичного:

| Тип | Что пересоздаётся | Когда |
|-----|-------------------|-------|
| Embedding-only | Только embeddings, не текст chunks | Смена embedding модели |
| Range rebuild | Chunks для диапазона ходов | Исправление factual error |
| Index-only | FTS5 + vector | Corrupted index |
| Full | Всё | Nuclear option |

## 3. Memory Scribe FSM

### 3.1. Обзор

Memory Scribe (§11.5 product-foundation.md) — это подсистема, которая после каждого хода предлагает обновления structured memory: новые факты, события, изменения отношений и т.д.

Ключевая проблема: LLM может ошибаться. Поэтому все предложения проходят через конечный автомат (FSM) с чётким жизненным циклом.

### 3.2. Диаграмма состояний

```
                    ┌─────────┐
                    │PROPOSED │ ← LLM предлагает запись
                    └────┬────┘
                         │
              ┌──────────┼──────────┐
              │          │          │
              ▼          ▼          ▼
        ┌──────────┐ ┌────────┐ ┌──────────┐
        │ APPROVED │ │REJECTED│ │  EDITED  │
        └────┬─────┘ └───┬────┘ └────┬─────┘
             │           │           │
             ▼           ▼           ▼
        ┌──────────┐ ┌───────────┐ ┌──────────┐
        │  ACTIVE  │ │BLACKLISTED│ │  ACTIVE  │
        └────┬─────┘ └───────────┘ └────┬─────┘
             │                          │
       ┌─────┼──────────┐         ┌─────┼──────────┐
       │     │          │         │     │          │
       ▼     ▼          ▼         ▼     ▼          ▼
  ┌────────┐┌────────┐┌────────┐  (аналогично)
  │SUPERSED││DISPUTED││ARCHIVED│
  └───┬────┘└───┬────┘└────────┘
      │         │
      ▼         ▼
 ┌────────┐┌──────────────┐
 │ARCHIVED││PENDING_REVIEW│
 └────────┘└──────┬───────┘
                  │
           ┌──────┼──────┐
           ▼      ▼      ▼
      ACTIVE  SUPERSEDED  REJECTED
```

### 3.3. Описание состояний

| Состояние | Описание | Попадает в контекст LLM? | Видимо пользователю? |
|-----------|----------|--------------------------|---------------------|
| `proposed` | LLM предложила, ожидает review | ❌ Нет | ✅ В review queue |
| `approved` | Пользователь подтвердил (транзитное) | ❌ Нет (переходит в active) | — |
| `edited` | Пользователь отредактировал (транзитное) | ❌ Нет (переходит в active) | — |
| `active` | Действующая запись в каноне | ✅ **Да** | ✅ В Memory Inspector |
| `rejected` | Пользователь отклонил | ❌ Нет | ✅ В rejected list |
| `blacklisted` | Отклонено + запрещено повторное предложение | ❌ Нет | ✅ В blacklist |
| `superseded` | Заменена новой версией | ❌ Нет | ✅ В истории записи |
| `disputed` | Обнаружено противоречие | ⚠️ С пометкой «спорный» | ✅ В review queue |
| `pending_review` | Возвращена на review (из disputed) | ❌ Нет | ✅ В review queue |
| `archived` | Больше не актуальна (но не ошибочна) | ❌ Нет | ✅ В архиве |

**Ключевое правило для Context Compiler:** только записи со статусом `active` попадают в контекст LLM как факты канона. Записи со статусом `disputed` могут попадать с явной пометкой «⚠ спорный факт — ожидает review».

### 3.4. Переходы состояний

```
proposed   → active          : user_approve
proposed   → active          : auto_approve (confidence >= threshold)
proposed   → edited → active : user_edit
proposed   → rejected        : user_reject
proposed   → blacklisted     : user_reject_and_blacklist

active     → superseded      : new_version_created
active     → disputed        : conflict_detected
active     → archived        : manual_archive / scene_ended / temporal_expiry
active     → pending_review  : user_flags_for_review

disputed   → active          : user_resolves (keeps this version)
disputed   → superseded      : user_resolves (picks other version)
disputed   → pending_review  : user_needs_more_info
disputed   → rejected        : user_rejects_both

rejected   → blacklisted     : user_blacklists
rejected   → proposed        : user_reconsiders (rare)

superseded → (terminal, no transitions out)
blacklisted→ (terminal, but user can remove from blacklist)
archived   → active          : user_reactivates (rare)
```

### 3.5. Blacklist: как предотвратить повторные предложения

**Проблема:** LLM может предложить тот же ошибочный факт снова и снова, потому что при следующем ходе он снова «видит» те же events.

**Механизм blacklist:**

1. При blacklist'е записи сохраняется её **семантическая сигнатура**:
   - subject + predicate + object (нормализованные)
   - Или embedding отклонённого текста

2. Перед отправкой предложений пользователю, Memory Scribe:
   - Сравнивает каждое новое предложение с blacklist
   - Если semantic similarity > threshold (например, 0.85) → предложение тихо отбрасывается
   - Логируется в debug log: «Отфильтровано по blacklist: {original} → {new_proposal}»

3. Blacklist привязан к scope:
   - Campaign-wide: «Лиара никогда не была в Городе Теней» (глобальный факт)
   - Scene-scoped: менее строгий, действует только в контексте сцены
   - Character-scoped: относится к конкретному персонажу

4. Пользователь может просматривать и очищать blacklist

### 3.6. Auto-approve

**Для продвинутых пользователей**, которые не хотят подтверждать каждый факт вручную.

#### 3.6.1. Порог confidence

```python
# Настройка в Campaign Settings
auto_approve_enabled: bool = False
auto_approve_threshold: float = 0.85  # 0.0 - 1.0
auto_approve_categories: list[str] = [
    "scene_update",      # обновления текущей сцены
    "character_location", # перемещения персонажей
    "minor_event",        # незначительные события
]
auto_approve_excluded_categories: list[str] = [
    "character_death",    # НИКОГДА не auto-approve
    "relationship_major", # крупные изменения отношений
    "secret_reveal",      # раскрытие секретов
    "canon_override",     # изменение установленного факта
]
```

#### 3.6.2. Confidence scoring

Confidence складывается из:

| Фактор | Влияние на confidence |
|--------|----------------------|
| Факт явно упомянут в тексте хода | +0.3 |
| Факт логически следует из действий | +0.2 |
| Согласуется с существующим каноном | +0.2 |
| Не противоречит ничему в active facts | +0.1 |
| Категория помечена как safe | +0.1 |
| Касается нового (не существующего) entity | +0.1 |

Пороги:
- < 0.5: не предлагать вовсе (скорее всего шум)
- 0.5-0.7: предложить, но не auto-approve
- 0.7-0.85: предложить, auto-approve если включено и категория разрешена
- \> 0.85: auto-approve (если включено)

#### 3.6.3. Auto-approve audit trail

Даже auto-approved записи логируются с пометкой `approved_by: "auto"`. Пользователь может:
- Просмотреть список auto-approved записей
- Отменить конкретные решения (перевести обратно в `proposed` или `rejected`)
- Настроить пороги на основе опыта

### 3.7. Batch Review UI

**Проблема:** после длинной игровой сессии (20-50 ходов) может накопиться 50-200 предложенных записей. Ревью по одной — утомительно.

**Решение: Batch Review интерфейс**

Элементы:
1. **Группировка по категориям:**
   - Факты (новые)
   - События
   - Изменения отношений
   - Обновления сцены
   - Изменения целей/намерений

2. **Массовые операции:**
   - «Одобрить все в категории»
   - «Одобрить все с confidence > X»
   - «Отклонить все в выделении»

3. **Quick-review карточки:**
   ```
   ┌─────────────────────────────────────────────┐
   │ 📝 ФАКТ                      confidence: 92%│
   │                                              │
   │ "Лиара получила письмо от Сафиры"           │
   │                                              │
   │ Источник: Ход #247                           │
   │ Категория: event                             │
   │                                              │
   │  [✓ Одобрить]  [✗ Отклонить]  [✏️ Править]  │
   └─────────────────────────────────────────────┘
   ```

4. **Diff view:** для записей, обновляющих существующие факты, показать old → new

5. **Keyboard shortcuts:** j/k для навигации, y/n для approve/reject, e для edit

### 3.8. Разрешение конфликтов

**Проблема:** два факта противоречат друг другу.

Примеры:
- Fact A: «Меч Пламени у Лиары» + Fact B: «Меч Пламени у стража»
- Fact A: «Король мёртв» + Fact B: «Король жив»
- Relationship A: «Лиара доверяет Сафире (0.8)» + Relationship B (новый): «Лиара не доверяет Сафире (0.3)»

#### 3.8.1. Обнаружение конфликтов

Continuity Checker (§11.3) выполняет проверки:

1. **Uniqueness constraints:** предмет не может быть у двух владельцев; персонаж не может быть в двух локациях
2. **Logical contradictions:** fact A и fact B имеют одинаковый subject+predicate, но разные object/value
3. **Temporal impossibility:** событие B произошло раньше события A, от которого зависит
4. **Semantic similarity + opposite polarity:** embedding-based detection противоречий

#### 3.8.2. Процесс разрешения

```
1. Обнаружен конфликт между Fact A (active) и Fact B (proposed)
2. Fact A переводится в состояние `disputed`
3. Fact B остаётся в `proposed`
4. Пользователю показывается:

   ⚠ КОНФЛИКТ ОБНАРУЖЕН
   
   Факт A (active с хода #102):
     "Меч Пламени находится у Лиары"
   
   Факт B (предложен на ходе #247):
     "Страж держит Меч Пламени"
   
   Варианты:
   [1] Сохранить A, отклонить B
   [2] Принять B, заменить A (A → superseded)
   [3] Отредактировать оба
   [4] Оставить оба (это допустимо, если персонажи
       могут иметь разные убеждения о местоположении меча)

5. Пользователь выбирает вариант
6. Записи переводятся в соответствующие состояния
```

#### 3.8.3. Автоматическое разрешение (без участия пользователя)

В некоторых случаях конфликт может быть разрешён автоматически:

| Ситуация | Автоматическое решение |
|----------|----------------------|
| Новый факт — обновление старого (тот же subject+predicate, позднее время) | Old → superseded, New → active |
| Факт помечен temporal interval, и старый истёк | Old → archived, New → active |
| Relationship update (тот же subject+object, новый intensity) | Old → superseded, New → active |

Автоматическое разрешение конфликтов **не должно** применяться к:
- Смерти/воскрешению персонажей
- Фундаментальным фактам мира
- Секретам и раскрытиям
- Любым записям с importance = Critical

## 4. Взаимодействие с другими подсистемами

### 4.1. Context Compiler

Context Compiler использует Memory Pipeline следующим образом:

1. Запрашивает **active** записи (facts, relationships, beliefs) для текущей сцены и персонажей
2. Запрашивает **memory chunks** через hybrid retrieval (§12 product-foundation.md)
3. Формирует **sliding window** из raw archive
4. Фильтрует всё по character knowledge scope (Belief → visibility)
5. Не включает `proposed`, `rejected`, `blacklisted`, `superseded`, `archived` записи

### 4.2. Continuity Checker

Continuity Checker потребляет `active` и `proposed` записи для поиска противоречий. При обнаружении — генерирует `dispute` event, который триггерит FSM transition.

### 4.3. Turn Runner

Turn Runner вызывает Memory Scribe после каждого хода. Порядок:

```
1. Turn Runner получает ответ от LLM
2. Turn Runner сохраняет Turn в raw archive
3. Turn Runner вызывает Memory Scribe
4. Memory Scribe анализирует Turn и генерирует proposed записи
5. Continuity Checker проверяет proposed на конфликты
6. Записи с конфликтами → disputed
7. Записи без конфликтов + auto_approve → active
8. Остальные → pending в review queue
9. UI обновляет review queue
```

## 5. Открытые вопросы по Memory Pipeline

1. **Суммаризация через LLM vs rule-based extraction:** для structured facts (entity+relation+entity) может быть эффективнее использовать structured extraction (JSON schema), а не freeform summarization. Как комбинировать?

2. **Версионирование chunks:** если пользователь исправил факт на ходе #50, а chunk покрывает ходы #40-60 — нужно ли пересоздать chunk? Или достаточно пометить факт как исправленный?

3. **Embedding drift:** при длинной кампании (>1000 ходов) ранние embeddings могут быть менее релевантны, если домен сместился (например, кампания началась в средневековье, а сейчас — в космосе). Нужна ли периодическая переиндексация?

4. **Multi-model summarization:** если пользователь использовал разные LLM в разные периоды, chunks будут иметь разное «качество» суммаризации. Это проблема?

5. **Privacy в chunks:** memory chunks могут содержать чувствительный контент (custom). Нужно ли шифрование at-rest?
